# 规则写了，Claude 就一定遵守吗？

你按前四篇的方法，搭了一套完整的规则体系——CLAUDE.md 主文件、6 份 docs/ 文档、frontmatter 规范、schema 定义、构建链路说明。

这套体系已经比大多数项目强了。但你很快会发现一个现实：

**规则再完善，Claude 也会犯错。**

不是因为它"不听话"，而是因为：

- 某些规则它确实看了，但在具体执行时遗漏了某一步
    
- 任务复杂时，它专注于主逻辑，忽略了边缘检查
    
- 你的规则覆盖了 90% 的场景，但这次刚好遇到了那 10%
    
- 规则写的是"不要做 X"，但 Claude 做了一个它没意识到等价于 X 的操作
    

这是所有 AI 协作的常态。即使是人类开发者，代码审查、单元测试、CI/CD 也不是因为"不信任开发者"，而是因为"人会犯错，系统来兜底"。

所以你需要的不只是"告诉 Claude 规则"，还需要"改完之后有一套机制来确认它做对了"。

这就是验证机制的作用。

# 内容型项目的验证和代码项目有什么不同

在代码项目中，验证的核心手段是自动化测试——单元测试、集成测试、E2E 测试。写一个函数，跑一遍测试套件，绿了就行。

内容型知识库项目的验证要复杂得多，原因有三：

**第一，很多问题不是"代码报错"，而是"数据不对"。**

frontmatter 缺一个字段，构建可能不报错——但列表页上这篇文章的摘要就是空的。category 拼错一个字，构建也不报错——但这篇文章从分类页上"消失"了。这类问题，自动化测试很难覆盖。

**第二，"看起来正常"不等于"真的正常"。**

详情页能打开，不代表搜索索引收录了这篇文章。列表页展示了这篇文章，不代表分类聚合页也展示了。你需要检查多个页面、多个环节，才能确认一次改动没有造成连锁问题。

**第三，改动的影响范围不直观。**

改一行代码，你大概知道影响哪几个函数。但改一篇内容的 category 字段，你不一定能立刻想到它会影响：分类聚合页的内容列表、侧边栏的分类计数、搜索索引中该文章的分类标记、可能存在的 RSS 分类过滤。

这三个特点决定了：内容型项目的验证策略，不能照搬代码项目的做法。

**你需要的是：自动化校验 + 手动检查清单的组合。**

# 三套检查清单

这是本篇最核心的交付物。三套清单分别应对三种场景：内容变更、结构变更、发布上线。

## 清单一：内容变更后检查清单

**触发场景**：新增一篇内容、修改一篇内容的正文或 frontmatter 字段、删除一篇内容。

这是最高频的场景。每次内容层面的改动，都应该走这套清单。

``## 内容变更后检查清单      ### 自动化检查（如项目支持）   - [ ] 运行 `npm run content:validate`（校验 frontmatter 完整性）   - [ ] 运行 `npm run content:build`（确认内容索引生成正常）   - [ ] 运行 `npm run lint` 和 `npm run type-check`（如改动涉及代码）      ### 手动检查   - [ ] 目标内容的详情页可正常打开   - [ ] 详情页的标题、摘要、日期显示正确   - [ ] 列表页中该内容出现且展示正常   - [ ] 分类聚合页中该内容出现在正确分类下   - [ ] 标签页中该内容出现在对应标签下（如适用）   - [ ] 搜索结果中可找到该内容（如项目有搜索功能）      ### 如果是修改已有内容   - [ ] slug 未被修改（除非明确被要求）   - [ ] category 和 tags 使用的是已有名称   - [ ] 未新增未定义的 frontmatter 字段      ### 如果是删除内容   - [ ] 确认无其他内容的 related_articles 引用了该 slug   - [ ] 确认无页面硬编码了该内容的链接   - [ ] 构建后无 404 页面   ``

**这套清单的设计逻辑**：先跑能自动化的，再人工看不能自动化的。自动化解决"结构对不对"，手动检查解决"效果对不对"。

## 清单二：结构变更后检查清单

**触发场景**：新增或修改 frontmatter 字段、修改内容解析逻辑、修改页面模板的数据消费方式、调整分类或标签体系、修改搜索索引生成逻辑。

结构变更比内容变更影响范围大得多。一次结构变更可能影响所有内容文件的处理方式。

`## 结构变更后检查清单      ### 类型和定义   - [ ] 类型定义已同步更新（如 types/content.ts）   - [ ] 新增/修改的字段在类型定义中有明确的类型标注      ### 解析和构建   - [ ] 内容解析函数正常工作   - [ ] 构建脚本正常执行（npm run build 无报错）   - [ ] 内容索引生成正常（npm run content:build 无报错）   - [ ] 搜索索引包含正确的字段      ### 页面消费   - [ ] 详情页正常渲染（抽查 3 篇不同类型的内容）   - [ ] 列表页正常展示   - [ ] 分类聚合页数据正确   - [ ] 标签聚合页数据正确      ### 兼容性   - [ ] 旧内容未因结构变更而失效   - [ ] 缺少新字段的旧内容有合理的默认值或降级处理   - [ ] draft 过滤逻辑仍然正常      ### 文档同步   - [ ] docs/data-schema.md 已更新   - [ ] docs/content-rules.md 已更新（如涉及 frontmatter 变更）   - [ ] docs/build-process.md 已更新（如涉及构建逻辑变更）   `

**这套清单的设计逻辑**：按影响链路逐层检查——类型定义 → 解析构建 → 页面消费 → 兼容性 → 文档同步。不放过任何一个环节。

注意最后的"文档同步"部分。这一步很多人会忘——改了代码但没更新文档，下次 Claude 读到的就是过期的规则。

## 清单三：发布前后检查清单

**触发场景**：准备将改动部署到正式环境。

`## 发布前检查清单      ### 内容层   - [ ] 所有新增/修改内容的 frontmatter 完整   - [ ] 无 slug 冲突   - [ ] category 和 tags 使用已有名称   - [ ] draft: true 的内容确认不应发布      ### 代码层   - [ ] lint 通过   - [ ] type-check 通过   - [ ] 构建成功   - [ ] 关键测试通过      ### 环境   - [ ] 环境变量齐全   - [ ] 无敏感信息提交      ---      ## 发布后检查清单      ### 核心页面   - [ ] 首页 / 主入口页正常加载   - [ ] 新增内容的详情页可访问   - [ ] 列表页展示正确      ### 聚合和搜索   - [ ] 分类聚合页数据正确   - [ ] 标签聚合页数据正确   - [ ] 搜索结果中可找到新内容（如适用）      ### 异常排查   - [ ] 无明显 404 页面   - [ ] 无空白页面或数据缺失   - [ ] 草稿内容未被公开发布   - [ ] 静态资源正常加载   `

**这套清单的设计逻辑**：发布前确认"能不能发"，发布后确认"发了之后有没有问题"。两段式设计，防止"构建通过了但线上出问题"的情况。

# 把三套清单放进正确的位置

清单设计好了，放哪里？

**docs/testing.md 中放完整版清单。** 包含每一条检查项的详细说明、适用场景和检查方法。

**CLAUDE.md 的 Workflows 中放精简版提醒。** 不需要把完整清单复制进去——只需要在每个 Workflow 的最后一步写一句"按 docs/testing.md 中的 XX 清单验证"。

比如第 2 篇写的"新增知识库内容"Workflow，最后一步本来是"检查详情页和列表页渲染是否正常"。现在可以改成：

``### 新增知识库内容   1. 搜索是否已有类似内容   2. 确认应放入哪个内容目录   3. 使用统一 frontmatter 结构   4. 文件名小写英文 + 中划线   5. category 和 tags 使用已有名称   6. 完成后按 `docs/testing.md` 的"内容变更后检查清单"验证   ``

这样做的好处：

- CLAUDE.md 保持精简（只多了一行引用）
    
- 完整的检查逻辑在 docs/testing.md 中维护（好更新、好扩展）
    
- Claude 每次做完任务都知道该去哪里找检查清单
    

# 哪些检查适合自动化，哪些只能手动

不是所有检查都能自动化。也不是所有检查都值得自动化。

## 适合自动化的检查

这些检查有一个共同特点：**规则明确、结果确定、可编程判断。**

**1. Frontmatter 完整性校验**

检查每篇内容文件是否包含所有必填字段，字段类型是否正确。

`// 伪代码示意   function validateFrontmatter(data: any): string[] {   const errors: string[] = [];   if (!data.title) errors.push('缺少 title');   if (!data.slug) errors.push('缺少 slug');   if (!data.category) errors.push('缺少 category');   if (!Array.isArray(data.tags)) errors.push('tags 必须是数组');   if (!data.date) errors.push('缺少 date');   if (typeof data.draft !== 'boolean') errors.push('draft 必须是布尔值');   return errors;   }   `

这种校验可以写成一个 npm script（如 `npm run content:validate`），每次构建前自动运行。

**2. Slug 唯一性检查**

扫描所有内容文件，确认没有两篇内容使用同一个 slug。

``// 伪代码示意   function checkSlugUniqueness(allContent: Content[]): string[] {   const slugMap = new Map<string, string[]>();   for (const item of allContent) {       const files = slugMap.get(item.slug) || [];       files.push(item.filePath);       slugMap.set(item.slug, files);     }   const duplicates: string[] = [];   for (const [slug, files] of slugMap) {       if (files.length > 1) {         duplicates.push(`slug "${slug}" 被多个文件使用: ${files.join(', ')}`);       }     }   return duplicates;   }   ``

**3. Category 合法性检查**

确认所有内容文件的 category 都在合法清单内。

``const VALID_CATEGORIES = ['政策解读', '备考指南', '考情速递', '常见问题'];      function checkCategory(allContent: Content[]): string[] {     return allContent       .filter(item => !VALID_CATEGORIES.includes(item.category))       .map(item => `${item.filePath}: category "${item.category}" 不在合法清单中`);   }   ``

**4. Date 格式检查**

确认所有日期字段使用统一格式。

**5. Draft 过滤验证**

在构建产物中检查，确认 `draft: true` 的内容没有出现在正式列表数据或搜索索引中。

## 这些自动化检查怎么组织

最推荐的方式是写一个统一的内容校验脚本：

`# package.json   {     "scripts": {       "content:validate": "ts-node scripts/validate-content.ts"     }   }   `

这个脚本一次性检查所有规则，输出清晰的错误报告。Claude 改完内容后运行一次，就能发现大部分结构性问题。

如果你还没有这样的脚本，可以作为一个后续任务——让 Claude 帮你写。在 CLAUDE.md 的 Commands 中预留这个命令，即使脚本还没实现，也能提醒 Claude"这类检查是需要做的"。

## 只能手动检查的项目

这些检查有一个共同特点：**需要"看效果"才能判断，无法用代码断言。**

**1. 页面渲染效果**

详情页打开后，标题显示对不对、排版乱不乱、图片加载正常吗？这些需要人眼看。

**2. 列表页展示效果**

新内容出现在列表里了吗？排序对吗？摘要显示完整吗？卡片样式正常吗？

**3. 聚合页数据正确性**

分类页里的内容数量对吗？有没有内容"消失"了？标签页呢？

**4. 搜索结果质量**

搜索某个关键词，目标内容出现了吗？排名合理吗？摘要片段有意义吗？

**5. 用户路径完整性**

从首页 → 分类页 → 详情页，链路是否通畅？面包屑导航正确吗？返回按钮能用吗？

## 手动检查的效率优化

手动检查听起来麻烦，但你可以通过两个方式提高效率：

**方式一：定义"标准检查路径"**

在 testing.md 中写清楚手动验证时应该走哪几个页面：

`## Manual Verification Path      内容变更后，按以下路径检查：   1. 打开目标内容的详情页 → 确认标题、摘要、正文、日期   2. 打开内容所属分类的聚合页 → 确认该内容出现在列表中   3. 打开任意标签页 → 确认标签关联正确   4. 使用搜索功能搜索内容标题的关键词 → 确认可被检索到   5. 返回首页 → 确认"最新内容"区域展示正常   `

有了固定路径，每次检查只需要 2-3 分钟，而不是漫无目的地点来点去。

**方式二：让 Claude 帮你做第一轮检查**

很多手动检查可以"半自动化"——你不用自己打开每个页面，而是让 Claude 在完成任务后报告检查结果：

在 Workflow 中加一步：

`6. 完成后，报告以下信息：      - 目标页面的 URL      - frontmatter 的完整内容      - 该内容的 category 和 tags      - 构建命令的执行结果   `

Claude 把这些信息汇报给你，你只需要扫一眼 frontmatter 对不对、打开一个 URL 确认页面正常——比从头排查快得多。

# 一个关键策略：按改动类型匹配验证力度

不是每次改动都需要跑完整检查清单。验证力度应该和改动的影响范围匹配。

`改动类型                     验证力度   ─────────────────────────────────────────   修改一篇文章的正文             低：详情页正常即可   修改一篇文章的 frontmatter     中：详情页 + 列表页 + 聚合页   修改分类或标签体系             高：全部聚合页 + 搜索   修改内容解析逻辑               高：抽查多篇 + 完整构建   修改构建脚本                   高：完整构建 + 全链路检查   新增内容类型                   最高：完整结构变更清单   `

在 testing.md 中写清楚这个对应关系，Claude 就能根据任务类型自动选择合适的验证力度，而不是每次都走最重的流程。

`## Verification Levels      ### Level 1：轻量验证（正文修改）   - 详情页渲染正常   - 构建无报错      ### Level 2：标准验证（frontmatter 修改）   - Level 1 全部项目   - 列表页展示正常   - 分类/标签聚合正常   - 搜索结果正常      ### Level 3：完整验证（结构变更）   - Level 2 全部项目   - 类型定义通过   - 抽查 3+ 篇不同类型内容   - 旧内容兼容性   - 文档同步更新      ### Level 4：全链路验证（构建或发布流程变更）   - Level 3 全部项目   - 完整构建测试   - 搜索索引完整性   - draft 过滤验证   - 部署前后检查清单   `

# 把验证嵌入 Workflow：不是"建议做"，而是"必须做"

前面讲的检查清单，如果只是放在 testing.md 里"以备查阅"，Claude 很可能不会主动去看。

验证要真正起作用，必须嵌入到 Claude 的工作流中——让它在完成任务的最后一步，自然而然地走验证流程。

## 方法一：在 Workflow 中显式包含验证步骤

每个 Workflow 的最后 1-2 步，应该是验证步骤：

``### 新增知识库内容   1. 搜索是否已有类似内容   2. 确认应放入哪个内容目录   3. 使用统一 frontmatter 结构   4. 文件名小写英文 + 中划线   5. category 和 tags 使用已有名称   6. 运行 `npm run content:build` 确认索引正常   7. 按 docs/testing.md Level 2 验证   ``

第 6 步是自动化验证（跑命令），第 7 步是对照清单做进一步检查。两步都在 Workflow 里，Claude 不会跳过。

## 方法二：在 Standards 中加入验证要求

``## Standards   - 任何内容变更完成后，至少运行 `npm run content:build` 确认构建正常   - 涉及 frontmatter 字段修改的任务，完成后需检查详情页和列表页   - 涉及数据结构或解析逻辑变更的任务，需按 docs/testing.md Level 3 验证   ``

这样验证就不只是 Workflow 里的建议，而是 Standards 里的"规则"。Claude 会更认真对待。

## 方法三：让 Claude 在完成时主动报告验证结果

在 CLAUDE.md 的 Notes 或 Standards 中加一条：

`- 完成任务后，简要报告验证结果：运行了哪些检查命令、结果如何、是否有需要注意的问题   `

这条规则让 Claude 不只是"做完就说完成了"，而是"做完后给你一份验证报告"。你可以快速扫一眼报告，判断是否需要进一步检查。

# 内容型项目最常见的 8 种验证失败

在你的 testing.md 中列出常见失败场景，可以帮助 Claude 在排查问题时更快定位原因。

`## Common Failure Cases      ### 1. 详情页 404   常见原因：   - slug 拼写错误   - 文件未放入正确的内容目录   - 路由配置未覆盖该内容类型      ### 2. 列表页不显示某篇内容   常见原因：   - frontmatter 缺少必填字段，解析时被跳过   - draft: true 未修改为 false   - date 格式错误，排序逻辑将其排到了不可见位置      ### 3. 分类聚合页缺少内容   常见原因：   - category 拼写不一致（如"备考指南"写成了"备考攻略"）   - 新分类没有对应的聚合页      ### 4. 搜索结果找不到某篇内容   常见原因：   - 搜索索引未重新生成（改完内容后没运行 content:build）   - draft: true 的内容被索引过滤   - 搜索索引脚本未收录目标字段      ### 5. 草稿内容被公开发布   常见原因：   - draft 过滤逻辑遗漏了某个查询入口   - 搜索索引未过滤 draft 内容   - RSS 或 sitemap 生成未过滤 draft      ### 6. 构建成功但页面数据不对   常见原因：   - frontmatter 字段有值但格式不对（如 tags 写成了字符串而非数组）   - 页面模板使用了错误的字段名   - 聚合逻辑的排序或过滤条件有误      ### 7. 新增字段后旧内容渲染异常   常见原因：   - 旧内容缺少新字段，渲染时未处理 undefined   - 类型定义标记为必填但旧数据没有该字段      ### 8. Slug 变更后出现大面积 404   常见原因：   - 其他内容的 related_articles 引用了旧 slug   - 搜索索引中仍是旧 slug   - 外部链接和收藏指向旧 URL   - 未设置 URL 重定向   `

这份"故障手册"的价值在于：Claude 遇到问题时，不需要从零开始猜测——它可以对照这 8 种常见场景逐一排查。

# 部署验证：构建通过不等于发布成功

很多开发者把"构建通过"等同于"可以发布"。

在内容型项目中，这个等式不成立。

构建通过只能证明：代码没有语法错误、类型检查通过、构建脚本能执行完。

构建通过不能证明：新内容出现在了正确位置、分类聚合数据正确、搜索索引包含了新内容、草稿没有被误发布。

这就是为什么你需要发布前后两套检查清单。

## 发布前：确认"能不能发"

发布前检查的核心问题是：**这次改动有没有遗漏、有没有风险、环境准备好了没有。**

重点项目：

- frontmatter 完整性（自动化检查可覆盖）
    
- slug 无冲突（自动化检查可覆盖）
    
- 草稿状态确认（人工确认 + 自动化辅助）
    
- 构建成功（自动化）
    
- 环境变量齐全（人工确认）
    

## 发布后：确认"发了之后没问题"

发布后检查的核心问题是：**用户能正常访问吗、内容展示正确吗、有没有意外的副作用。**

重点项目：

- 核心页面可访问（手动检查）
    
- 新内容在列表页和详情页展示正常（手动检查）
    
- 聚合和搜索正常（手动检查）
    
- 无 404 和空页面（手动 + 自动化辅助）
    
- 草稿未被公开（手动检查）
    

## 在 deployment.md 中怎么写

发布检查清单应该放在 docs/deployment.md 中，紧接在"标准发布流程"之后。

同时在 CLAUDE.md 中可以加一条 Standard：

`- 准备发布时，按 docs/deployment.md 的发布前后清单执行检查   `

# 测试数据的管理

如果你要给内容校验写自动化测试，你需要测试数据。

## 推荐做法

在 `tests/fixtures/` 或 `tests/content-samples/` 中维护一组样本内容文件：

`tests/fixtures/content/   ├── valid-article.md           # 完整正确的文章   ├── valid-faq.md               # 完整正确的 FAQ   ├── missing-slug.md            # 缺少 slug 的文件（用于测试校验逻辑）   ├── invalid-category.md        # category 不在合法清单中   ├── duplicate-slug.md          # 与 valid-article.md slug 相同   ├── draft-article.md           # draft: true 的文章   └── missing-frontmatter.md     # 完全没有 frontmatter   `

这组样本文件覆盖了"正确"和"各种错误"的情况。你的校验脚本可以用它们来测试自身的正确性。

## 测试数据的原则

- 使用稳定的固定样本，不依赖真实内容数据
    
- 覆盖典型正确情况和典型错误情况
    
- 保持样本简短——只需要能触发目标检查逻辑的最小内容
    
- 新增一种校验规则时，同步新增对应的测试样本
    

# 把验证能力写进 docs/testing.md 的完整结构

把前面所有内容组织起来，一份完整的 docs/testing.md 应该包含以下结构：

`# Testing Guide      ## Purpose   本文档说明内容型知识库项目中的验证方式和检查清单。      ## Verification Levels   ### Level 1：轻量验证   ### Level 2：标准验证   ### Level 3：完整验证   ### Level 4：全链路验证      ## 内容变更后检查清单      ## 结构变更后检查清单      ## 发布前检查清单      ## 发布后检查清单      ## Automated Checks   ### frontmatter 校验   ### slug 唯一性检查   ### category 合法性检查      ## Manual Verification Path      ## Common Failure Cases      ## Test Data Rules   `

每个模块的内容，就是本篇前面各节讲的那些。

# 一个容易被忽视的验证场景：跨内容影响

大多数检查清单关注的是"我改的这篇内容本身有没有问题"。但有一类问题更隐蔽：**改了 A 内容，影响了 B 内容。**

常见场景：

**修改了分类名称，导致同分类下其他内容的聚合页受影响。**

你把一篇文章从"政策解读"改成了"考情速递"。这篇文章本身没问题。但"政策解读"分类页少了一篇内容、"考情速递"分类页多了一篇——如果这两个分类页有"内容数量"展示或分页逻辑，它们都会受影响。

**删除了一篇内容，但其他内容的 related_articles 还引用着它的 slug。**

被引用的内容没了，引用方的"相关推荐"区域可能显示空白或报错。

**修改了 Tag 名称，导致同标签下其他内容的标签聚合受影响。**

你把一个标签从"教资"改成了"教师资格证"。改的那篇没问题，但其他用了"教资"标签的文章，在新标签页里就找不到了。

**怎么防范：**

在 testing.md 中加一条提醒：

`## Cross-content Impact      修改 category、tags 或删除内容时，需额外检查：   - 同分类下的其他内容是否受影响   - 同标签下的其他内容是否受影响   - 是否有其他内容的 related_articles 引用了被修改/删除的 slug   - 聚合页的内容数量是否发生了预期外的变化   `

# 验证的长期演进

验证体系不是一次建好就不动了。随着项目发展，它应该持续演进：

**第一阶段：只有手动清单**

项目早期，自动化校验脚本还没写，先用手动检查清单兜底。Claude 每次做完任务对照清单检查。

**第二阶段：核心检查自动化**

写一个 `content:validate` 脚本，覆盖 frontmatter 完整性、slug 唯一性、category 合法性。手动清单只保留"看页面效果"的部分。

**第三阶段：自动化 + hooks**

把 `content:validate` 接入 Claude Code hooks，每次 Claude 修改内容文件后自动运行。不需要 Claude "记住"去跑校验——hook 替它执行。

第三阶段是下一篇的内容。

# 本篇核心认知回顾

**认知一：规则和验证是互补的**

规则告诉 Claude "什么该做"，验证确认"它做对了没有"。只有规则没有验证，等于没有质量兜底。

**认知二：内容型项目的验证 = 自动化校验 + 手动检查**

结构性问题（字段缺失、slug 冲突、category 非法）适合自动化。效果性问题（页面渲染、聚合正确性、搜索质量）需要手动检查。

**认知三：三套清单覆盖三种场景**

内容变更 → 标准验证。结构变更 → 完整验证。发布上线 → 前后两段验证。

**认知四：验证力度应该匹配改动影响范围**

不是每次都跑最重的清单。改一行正文和改整个数据结构，需要的验证力度完全不同。

**认知五：验证必须嵌入 Workflow，而不是另存参考**

在 Workflow 最后一步显式写验证要求，在 Standards 中把验证定为规则。让验证成为 Claude 工作流的一部分，而不是可选的额外动作。

**认知六："构建通过"不等于"发布安全"**

内容型项目有很多问题在构建阶段不会暴露，只有在页面渲染时才能发现。发布后检查不是多余的。
