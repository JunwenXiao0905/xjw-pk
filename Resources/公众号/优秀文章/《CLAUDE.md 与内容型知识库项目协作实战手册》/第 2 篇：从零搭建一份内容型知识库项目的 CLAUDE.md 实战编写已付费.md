> 📌 本篇核心目标：从空白文件开始，手把手写出一份适用于内容型知识库项目的 CLAUDE.md。你会得到两份成品模板（最小可用版 + 进阶增强版），以及每个模块"为什么这样写"的详细拆解。

# 先别急着写，搞清楚三种创建方式

在动手之前，你需要知道 CLAUDE.md 有三种创建方式，各有适用场景。

## 方式一：/init 命令自动生成

打开你的项目目录，启动 Claude Code，输入 `/init`：

`cd your-project   claude   /init   `

Claude 会扫描你的项目——看 package.json、tsconfig、目录结构、现有文档——然后自动生成一份 CLAUDE.md 初稿。

生成的内容通常包括：构建命令、测试命令、关键目录、它检测到的代码约定。

**优点**：快，有 baseline，对新接手的项目很友好。

**问题**：自动生成的内容往往偏冗余。它会把能检测到的东西都塞进去，不会帮你做"这条规则重不重要"的筛选。而且它几乎不可能自动识别内容型项目的特殊规则——比如 frontmatter 约束、slug 规则、分类体系的设计逻辑。

**正确用法**：把 `/init` 当草稿，不要当成品。生成后必须人工精修——删掉冗余、补充它遗漏的关键规则、调整结构。

如果你的项目已经有了 CLAUDE.md，也可以再跑一次 `/init`。Claude 会基于对项目的重新扫描，给出改进建议。

## 方式二：手动从零创建

在项目根目录创建文件：

`touch CLAUDE.md   `

然后用任何 Markdown 编辑器打开，从空白开始写。

**适合**：对项目很熟悉的人、想完全掌控内容的人、不想引入自动生成冗余的人。

**这也是本篇主要教你的方式。**

## 方式三：# Memory 快捷添加

在 Claude Code 对话中，输入 `#` 加上一条规则：

`# 新增内容必须使用统一的 frontmatter 模板   `

Claude 会把这条规则追加到 CLAUDE.md 文件中。

**适合**：工作过程中随手补充。当你发现"Claude 又犯了这个错"或者"这个约定我每次都要重复说"时，立刻用 `#` 记录下来。

**最佳实践**：不要试图用 `#` 从零构建一份完整的 CLAUDE.md——它会变成一堆无序的规则碎片。`#` 的价值在于"持续迭代"，而不是"初始创建"。

## 我的建议

组合使用：

`第 1 步：手动创建骨架（或用 /init 生成草稿后精修）   第 2 步：实际协作 3-5 次，发现遗漏   第 3 步：用 # 快速补充新规则   第 4 步：每隔一段时间回来整理一次   `

最好的 CLAUDE.md 不是一次写完的，而是在真实使用中逐步长出来的。

但你需要一个好的骨架起步。接下来就教你搭这个骨架。

# 核心骨架：7 个必备模块

不管你的项目多大多小，一份 CLAUDE.md 最核心的骨架就是 7 个模块：

`# Project Context          ← 项目是什么 + 协作哲学   ## About This Project       ← 技术栈和架构   ## Key Directories          ← 关键目录   ## Commands                 ← 常用命令   ## Standards                ← 关键规则   ## Workflows                ← 高频任务步骤   ## Notes                    ← 防坑提醒   `

这 7 个模块不是我随意拍的，而是从"Claude 做事前必须知道什么"倒推出来的：

- 它需要知道**项目是什么**（Project Context + About）
    
- 它需要知道**代码在哪里**（Key Directories）
    
- 它需要知道**怎么运行项目**（Commands）
    
- 它需要知道**什么不能做**（Standards）
    
- 它需要知道**做事的正确顺序**（Workflows）
    
- 它需要知道**哪里有坑**（Notes）
    

下面逐个拆解怎么写。

# 模块一：Project Context

这是 CLAUDE.md 的第一段，Claude 进入项目后最先看到的内容。

## 它的任务

用最短的篇幅告诉 Claude 两件事：

1. 这个项目是做什么的
    
2. 你希望它以什么态度和原则来工作
    

## 怎么写

**项目说明只写一句话。** 不要写项目历史、团队背景、用户画像。一句话说清"这是什么类型的项目，核心目标是什么"就够了。

**协作哲学写一到两条。** 这是你给 Claude 设定的"做事态度"。

## 示例

`# Project Context      教师考试知识库系统。Markdown 驱动的内容型静态站点，面向备考用户提供考情、政策和备考指南。   处理任务时，优先理解现有内容结构和字段设计，再做增量式修改；避免随意重构。   `

## 常见错误

`# ❌ 太长，像在写项目介绍书      # Project Context      本项目是一个面向全国中小学教师的综合性知识库系统，旨在为正在备考教师资格证、   教师编制、特岗教师招聘的用户提供一站式信息服务。项目始于 2024 年 3 月，   由三人团队协作开发。目前已积累超过 500 篇原创内容，覆盖 12 个省份的   考试信息。项目采用 JAMStack 架构理念，追求高性能、低成本和优秀的 SEO 表现……   `

Claude 不需要知道你团队有几个人、项目什么时候开始、积累了多少篇文章。这些信息不会影响它做任何具体任务。

# 模块二：About This Project

## 它的任务

告诉 Claude 你用了什么技术，项目大概是什么架构。

## 怎么写

列出框架、语言、关键依赖。如果有内容处理的特殊机制（比如 Markdown 解析、搜索索引生成），在这里提一句。

不需要列出 package.json 里的每一个依赖。只列 Claude 在做任务时真正需要知道的。

## 示例

``## About This Project   - 框架：Next.js 14 + TypeScript   - 内容源：`content/` 目录下的 Markdown 文件，通过 frontmatter 定义元数据   - 内容解析：自定义解析函数，位于 `lib/content.ts`   - 搜索：构建时生成搜索索引，脚本位于 `scripts/build-search-index.ts`   - 样式：Tailwind CSS   - 部署：Vercel   ``

## 写法要领

**内容型项目有一条额外信息必须写**：内容是怎么进入站点的。

普通代码项目不需要解释"代码怎么变成功能"——这是编译器的事。但内容型项目的数据链路不是显而易见的：内容文件在哪里 → 怎么被解析 → 怎么变成页面数据 → 怎么被搜索索引收录。

你不需要在这里把整个链路写完（那是 docs/build-process.md 的工作），但至少要提一句"内容源在哪、用什么解析"。这样 Claude 在碰到内容相关任务时，知道该去哪里找代码。

# 模块三：Key Directories

## 它的任务

让 Claude 快速建立目录地图，知道什么代码在哪里。

## 怎么写

**不要列出所有目录。** 只列两类：

1. **高价值目录**：Claude 做任务时经常需要访问的
    
2. **高风险目录**：Claude 容易放错文件或做错操作的
    

## 示例

``## Key Directories   - `content/articles/`：知识库文章（Markdown + frontmatter）   - `content/faq/`：常见问题   - `content/guides/`：操作指南   - `app/`：页面路由与入口   - `components/`：通用 UI 组件   - `lib/`：内容解析、工具函数   - `scripts/`：构建脚本（搜索索引、内容预处理）   - `generated/`：构建生成的中间文件，不要手动修改   - `public/`：静态资源      > 新增代码优先放入已有目录，不随意创建新的顶层目录。   > `generated/` 目录的文件由脚本自动生成，任何手动修改都会在下次构建时被覆盖。   ``

## 写法要领

注意最后那两行附注——这不是可有可无的提醒，而是防止 Claude 犯两个常见错误：

1. Claude 经常自作主张创建新目录。比如你让它加一个工具函数，它可能不放 `lib/`，而是创建一个全新的 `utils/` 目录。写一句"优先放入已有目录"就能大幅减少这种情况。
    
2. `generated/` 这类构建产物目录，Claude 不知道它是自动生成的，可能会直接去改里面的文件。显式标注"不要手动修改"可以避免无意义的修改。
    

**内容型项目特别需要把内容目录写清楚。** 如果你的 `content/` 下有多个子目录（articles、faq、guides、topics），最好每个都列出来，标注用途。因为 Claude 新增内容时需要知道该放哪个目录。

# 模块四：Commands

## 它的任务

告诉 Claude 怎么运行项目、怎么测试、怎么构建。

## 怎么写

分组列出，每条命令附上一句说明。

## 示例

``## Commands      ### Development   - `npm install`：安装依赖   - `npm run dev`：启动本地开发环境   - `npm run build`：完整构建项目   - `npm run preview`：预览构建结果      ### Code Quality   - `npm run lint`：代码检查   - `npm run type-check`：类型检查      ### Content   - `npm run content:build`：生成内容索引和搜索数据   - `npm run content:validate`：校验所有内容文件的 frontmatter（如有此命令）      > 修改内容文件后，建议运行 `npm run content:build` 确认索引生成正常。   > 提交前至少运行 `npm run lint` 和 `npm run type-check`。   ``

## 写法要领

**内容型项目通常有"内容构建"这个独立环节。** 很多内容站点的构建分两步：先处理内容（生成索引、解析 frontmatter、构建搜索数据），再构建站点。如果你的项目有这类命令，一定要单独列出来。

Claude 经常会在改完内容后直接说"完成了"，而忘记运行内容构建。在这里写上"修改内容后建议运行 xxx"可以提醒它。

**不要写太多命令。** 只写 Claude 在日常协作中真正会用到的。什么 `npm run analyze-bundle`、`npm run storybook` 之类偶尔才用的，不需要出现在主文件里。

# 模块五：Standards

## 它的任务

告诉 Claude 在这个项目里"什么不能做"和"什么必须做"。

## 这是最容易写废的一个模块

Standards 写不好，整个 CLAUDE.md 就废了一半。因为大部分开发者在这个模块犯同一个错误：**把所有能想到的规范全部堆进去。**

`# ❌ 典型的错误写法      ## Standards   - 使用 2 空格缩进   - 字符串用单引号   - 语句末尾加分号   - 数组末尾加逗号   - 回调函数用箭头函数   - import 按字母排序   - 组件名用 PascalCase   - 工具函数用 camelCase   - 常量用 UPPER_SNAKE_CASE   - CSS 类名用 kebab-case   - 文件名用 kebab-case   - 每个函数都要有类型标注   - 不要用 any   - 不要用 var   - React 组件必须是函数组件   - 不要用 class 组件   - ……还有 40 条   `

这份清单有什么问题？

**每一条都是对的，但它们不应该出现在 CLAUDE.md 里。**

原因有三：

1. **这些是 linter 的活。** ESLint + Prettier 可以自动检查并修复这些问题。用提示词来约束代码风格，是用错了工具。
    
2. **稀释了真正重要的规则。** 当 Standards 里有 50 条规则时，Claude 对每条规则的遵守力度都会下降。那些真正关键的——比如"不要改 slug"——被淹没在一堆格式规范里。
    
3. **Claude 大部分时候能自动遵守这些。** 如果你的项目已有一致的代码风格，Claude 会从现有代码中推断出来。你不需要显式告诉它"用 2 空格缩进"——它看到你的代码用 2 空格，自然会跟着用。
    

## 正确的写法原则

Standards 里只放符合这个条件的规则：

> **如果 Claude 不知道这条规则，它大概率会做错，而且做错后的修复成本很高。**

## 示例

`## Standards      ### 内容规范   - 所有内容文件必须包含完整的 frontmatter（title, slug, category, tags, date, draft）   - 不要修改已发布内容的 slug——slug 变更会导致 URL 断裂、搜索索引失效、内链 404   - 不要随意新增 frontmatter 字段，新字段需要确认页面模板和构建脚本是否支持   - category 必须使用已有的分类名称，不要发明近义词   - tags 命名保持一致，不混用全称和缩写      ### 代码规范   - 优先复用已有组件和工具函数，不重复造轮子   - 不要无理由重命名文件、函数或目录   - 涉及公共接口或导出结构的修改，先确认影响范围   - 新增代码与当前项目风格保持一致      ### 安全规范   - 不提交 .env 文件、API 密钥、Token   - 不直接操作生产环境配置   `

对比一下：这份 Standards 只有 11 条。但每一条都指向一个"做错后代价很大"的场景。

前 5 条是**内容型项目独有的**，也是这本手册强调的重点。普通代码项目不需要这些——但在内容项目里，这 5 条规则的遵守程度直接决定了你的站点结构是否会越来越乱。

后面的代码规范和安全规范是通用的，但也只挑了最关键的几条。

## 一个辅助判断方法

写完每一条 Standard 后，问自己一个问题：

> "如果我不写这条，Claude 有多大概率会自己做对？"

如果答案是"大概率能做对"，就不需要写。

如果答案是"它根本不可能知道这个规则"——比如"category 必须使用已有名称"——那这条就必须写。

# 模块六：Workflows

## 它的任务

告诉 Claude 做高频任务时的正确步骤顺序。

## 为什么 Workflows 比 Standards 更重要

Standards 告诉 Claude "规则是什么"。但 Claude 有时候知道规则，却不知道正确的做事顺序。

比如你让 Claude "给知识库新增一篇 FAQ"。没有 Workflow 的时候，Claude 可能直接开始写一个 Markdown 文件——内容写得不错，但它跳过了几个关键步骤：

- 没有检查是否已有类似内容
    
- 没有使用项目统一的 frontmatter 模板
    
- 放错了目录
    
- 写完之后没有运行内容构建验证
    

有了 Workflow，Claude 会按你定义的顺序一步一步来。

## 示例

``## Workflows      ### 新增知识库内容   1. 先搜索项目中是否已有类似内容，避免重复   2. 确认内容应放入哪个目录（articles / faq / guides）   3. 使用统一的 frontmatter 结构（参考现有内容文件）   4. 文件名使用小写英文 + 中划线，与 slug 保持一致   5. 确保 category 使用已有分类名，tags 使用已有标签   6. 完成后运行 `npm run content:build` 验证索引生成   7. 检查详情页和列表页是否正常渲染      ### 修改已有内容   1. 先阅读原文件的 frontmatter 和正文结构   2. 确认修改范围——是只改正文，还是涉及字段变更   3. 如果修改 category 或 tags，确认是否影响聚合页   4. 不要修改 slug，除非明确被要求且已评估影响   5. 完成后运行 `npm run content:build` 验证   6. 检查详情页渲染是否正常      ### 修改内容解析逻辑或页面模板   1. 先理解现有的数据流：内容文件 → 解析函数 → 页面组件   2. 确认修改是在哪个环节：内容层、解析层、还是渲染层   3. 小范围修改，先验证单篇内容   4. 再运行完整构建，确认不影响其他内容   5. 检查列表页、详情页、搜索是否正常   ``

## 写法要领

**每个 Workflow 控制在 5-7 步。** 太短没有约束力，太长 Claude 容易跳步。

**步骤顺序不是随便排的。** 注意每个 Workflow 的前几步都是"先看、先确认"，而不是"直接动手"。这是刻意设计的——Claude 最大的问题就是"跳过理解直接执行"，Workflow 的作用就是强制它先看再做。

**只写高频任务。** 不需要给每种可能的任务都写 Workflow。只写你和 Claude 协作中最常做的 2-4 类任务。偶尔做的任务，放到 docs/ 文档里或者在对话中临时交代。

**内容型项目最推荐的三个 Workflow 就是上面这三个：** 新增内容、修改内容、修改内容处理逻辑。它们覆盖了 80% 的日常协作场景。

# 模块七：Notes

## 它的任务

放不属于其他模块、但 Claude 必须知道的注意事项。可以理解为"防坑区"。

## 示例

``## Notes   - 不要提交敏感信息（.env、密钥、Token、数据库连接）   - `generated/` 目录的文件由构建脚本自动生成，不要手动修改   - 若发现内容文件的 frontmatter 与项目规范不一致，优先记录问题，不要大批量修改   - 若索引文件或文档与代码实现不一致，以当前代码为准   - 复杂任务优先拆解为多个小步骤，不直接进行大范围改动   ``

## 写法要领

Notes 是一个"兜底模块"——那些你觉得重要但塞不进其他模块的信息，放这里。

**不要让 Notes 膨胀。** 如果你的 Notes 超过了 10 条，说明有些内容应该移到 Standards 或 Workflows 中，或者拆到 docs/ 文档里。

Notes 最适合放三类信息：

1. **安全提醒**：敏感信息、生产环境操作
    
2. **容易误操作的边界**：自动生成目录、只读文件、共享配置
    
3. **协作策略**：遇到不一致怎么办、遇到复杂任务怎么办
    

# 把 7 个模块组装起来：最小可用版

现在你已经理解了每个模块的写法。把它们组合起来，就是一份完整的 CLAUDE.md。

下面是一个**最小可用版模板**——适合你刚开始搭建时使用。它足够精简，但覆盖了 Claude 最需要知道的所有关键信息。

``# Project Context      本项目是一个【项目类型描述】，主要用于【核心目标】。   处理任务时，优先理解现有结构和字段设计，再做增量修改。不随意重构。      ## About This Project   - 框架：【框架名称 + 语言】   - 内容源：`content/` 目录下的 Markdown 文件，通过 frontmatter 定义元数据   - 内容解析：【解析方式和关键文件位置】   - 搜索：【搜索实现方式，如有】   - 样式：【CSS 方案】   - 部署：【部署平台】      ## Key Directories   - `content/articles/`：【说明】   - `content/faq/`：【说明】   - `content/guides/`：【说明】   - `app/`：页面路由与入口   - `components/`：通用 UI 组件   - `lib/`：内容解析、工具函数   - `scripts/`：构建脚本      > 新增代码优先放入已有目录，不随意创建新的顶层目录。      ## Commands   - `npm install`：安装依赖   - `npm run dev`：启动开发环境   - `npm run build`：完整构建   - `npm run lint`：代码检查   - `npm run type-check`：类型检查   - `npm run content:build`：生成内容索引（如有此命令）      > 修改内容文件后，建议运行内容构建命令确认索引正常。      ## Standards   - 所有内容文件必须包含完整 frontmatter（title, slug, category, tags, date, draft）   - 不要修改已发布内容的 slug   - 不要随意新增 frontmatter 字段   - category 使用已有分类名，不发明近义词   - tags 命名保持一致   - 优先复用已有组件和函数   - 不提交 .env、密钥、Token      ## Workflows      ### 新增内容   1. 搜索是否已有类似内容   2. 确认应放入哪个内容目录   3. 使用统一的 frontmatter 结构   4. 文件名小写英文 + 中划线   5. 完成后运行内容构建验证      ### 修改已有内容   1. 先阅读原文件 frontmatter 和结构   2. 确认修改范围   3. 不修改 slug，除非被明确要求   4. 完成后运行内容构建验证      ## Notes   - `generated/` 目录文件由脚本生成，不手动修改   - frontmatter 与规范不一致时，记录问题而非大批量修改   - 文档与代码不一致时，以代码为准   ``

**这份模板大约 50 条有效信息。** 加上 Claude Code 自身的系统指令约 50 条，总计约 100 条——远在安全范围内，Claude 可以稳定遵守。

# 进阶增强版：适合成熟项目

如果你的项目已经比较成熟，或者你想一步到位，可以在最小版基础上增加以下模块：

``## Conditional Rules      ### 在 content/ 目录下工作时   - 保持 frontmatter 字段一致   - 不随意发明新内容格式   - 优先复用已有内容模板      ### 在 tests/ 目录下工作时   - 优先复用已有 fixtures   - 测试结构尽量与源文件对应   - 不依赖真实外部服务，使用 mock      ### 在 scripts/ 目录下工作时   - 脚本应可重复执行   - 破坏性操作需提供明确提示   - 修改后检查是否影响内容构建流程      ## Additional Documentation      处理以下任务前，请优先查看对应文档：   - 内容结构与 frontmatter：`docs/content-rules.md`   - 字段与数据实体：`docs/data-schema.md`   - 构建与索引流程：`docs/build-process.md`   - 验证与测试：`docs/testing.md`   - 发布与上线：`docs/deployment.md`   - 项目实现约定：`docs/code-conventions.md`      仅在任务相关时读取对应文档，不必每次全部加载。      ## File Boundaries   - 非必要不修改全局配置文件   - 非必要不批量重命名文件   - 修改公共组件或基础工具层时，需特别注意影响范围   - 对导出结构、数据格式的修改需谨慎   ``

增强版比最小版多了三个模块：

**Conditional Rules**：按目录定义差异化规则。当 Claude 在不同目录下工作时，自动适用不同的约束。这在 Monorepo 或多目录结构的项目中特别有用。

**Additional Documentation**：指向 docs/ 目录下的专项文档。这是"渐进披露"策略的入口——第 3 篇会详细讲怎么设计这套文档体系。

**File Boundaries**：显式标注修改边界。防止 Claude 在做局部任务时"顺手"改了不该改的东西。

# 内容型项目的 5 条专属规则

到这里，你已经掌握了 CLAUDE.md 的通用写法。但内容型知识库项目还有一些**专属的**规则需要特别关注。

这 5 条规则，在普通代码项目中不需要，但在内容型项目中缺一不可：

## 规则 1：显式声明 frontmatter 必填字段

`所有内容文件必须包含完整 frontmatter：title, slug, category, tags, date, draft   `

为什么这条必须写？因为 Claude 不会自动知道你的 frontmatter 需要哪些字段。如果你不告诉它，它可能只写一个 `title`，甚至不写 frontmatter。

## 规则 2：slug 保护规则

`不要修改已发布内容的 slug——slug 变更会导致 URL 断裂、搜索索引失效、内链 404   `

为什么要写这么详细的后果说明？因为 Claude 需要理解**为什么**不能改 slug，而不只是记住"不能改"。当它理解了后果的严重性，在面对模糊场景时更不容易违反这条规则。

比如你说"帮我优化一下这篇文章"，如果 Claude 只记住"不改 slug"，但不理解原因，它可能会想"优化 slug 也是优化的一部分吧"。但如果它知道改 slug 等于"URL 断裂 + 搜索失效 + 内链 404"，它就不会轻易触碰。

## 规则 3：分类和标签的一致性约束

`category 必须使用已有的分类名称，不要发明近义词   tags 命名保持一致，不混用全称和缩写   `

这两条看起来简单，但它们防住的是一个非常隐蔽的问题：聚合失真。

假设你的分类页是按 `category` 精确匹配来聚合内容的。如果 Claude 把一篇新文章的 category 写成"政策分析"而不是你已有的"政策解读"，这篇文章就会从分类页上"消失"。页面不报错，构建不报错，但用户就是找不到这篇内容。

这种问题排查起来非常痛苦。在 CLAUDE.md 里加这两条规则，可以从源头预防。

## 规则 4：新增字段的约束

`不要随意新增 frontmatter 字段，新字段需要确认页面模板和构建脚本是否支持   `

Claude 对 Markdown 文件非常"有创意"。你让它"给文章加个摘要"，它可能不用你已有的 `description` 字段，而是发明一个 `summary` 字段。你让它"标记推荐文章"，它可能创造一个 `recommended: true`，而你的项目根本没有处理这个字段的逻辑。

这条规则让 Claude 在新增字段之前先想一下"这个字段有人消费吗"。

## 规则 5：内容修改后的验证提醒

``修改内容文件后，建议运行 `npm run content:build` 确认索引生成正常。   ``

在代码项目中，改完代码会自动触发热更新或编译检查，反馈是即时的。但在内容项目中，改完一篇 Markdown 文件，你看不到任何即时反馈——只有在构建时才会发现问题。

这条规则提醒 Claude：改完内容不是"完成了"，还需要验证。

# 常见误区一览

写 CLAUDE.md 时最常见的 6 个误区，附上修正方式：

## 误区 1：把它当 linter

`# ❌ 不要这样做   - 用 2 空格缩进   - 用单引号   - 加分号   - 数组尾逗号   - 箭头函数   - ……   `

**修正**：这些交给 ESLint / Prettier。在 CLAUDE.md 里只需要一句："提交前运行 `npm run lint`"。

如果你想更进一步，可以设置一个 pre-commit hook 或 Claude Code hook，让 lint 自动运行。第 6 篇会详细讲。

## 误区 2：写太长的项目介绍

`# ❌ 不要这样做   本项目始于 2024 年 3 月，由产品经理小王提出需求，最初是一个内部工具……   `

**修正**：一句话说清项目是什么就够了。Claude 不需要知道项目历史。

## 误区 3：列出所有目录

`# ❌ 不要这样做   - src/   - src/components/   - src/components/ui/   - src/components/layout/   - src/components/common/   - src/lib/   - src/lib/utils/   - src/lib/hooks/   - ……列出了 30 个目录   `

**修正**：只列高价值和高风险目录。Claude 可以自己用 `ls` 看目录结构。

## 误区 4：Standards 里放了不稳定的规则

`# ❌ 不要这样做   ## Standards   - 本周先不要改 payment 模块，下周重构   - 登录页正在 A/B 测试，不要改样式   `

**修正**：临时性的规则在对话中交代就行，不要写进 CLAUDE.md。主文件只放长期有效的规则。

## 误区 5：Workflows 写成了教学文档

`# ❌ 不要这样做   ### 新增 API 端点   首先，我们需要了解 RESTful API 的设计原则。REST 是一种……   然后，打开 src/api 目录。你会看到目录中有多个文件，它们分别负责……   `

**修正**：Workflows 是给 Claude 看的步骤清单，不是给人看的教程。写成编号步骤，每步一句话。

## 误区 6：忘了 CLAUDE.md 需要持续维护

很多人第一天认真写了一份 CLAUDE.md，然后就再也没动过。三个月后，里面的命令已经过时了、目录结构变了、工作流也不同了。

**修正**：在 PR 检查清单里加一条"CLAUDE.md 是否需要更新"。用 `#` 快捷方式随时补充新规则。每隔一两个月花 10 分钟清理一次。

# 一个完整的成品示例

最后，给你一份**真实场景的完整示例**——一个教师考试知识库项目的 CLAUDE.md。

你可以把它当作参照物，对着改成你自己项目的版本。

``# Project Context      教师考试知识库系统。Markdown 驱动的内容型站点，面向备考用户提供考情查询、政策解读和备考指南。   处理任务时，优先理解现有内容结构和字段设计，再做增量修改。避免随意重构。      ## About This Project   - 框架：Next.js 14 + TypeScript   - 内容源：`content/` 目录下的 Markdown 文件，通过 frontmatter 定义元数据   - 内容解析：`lib/content.ts` 中的解析函数   - 搜索：构建时通过 `scripts/build-search-index.ts` 生成搜索索引   - 样式：Tailwind CSS   - 部署：Vercel      ## Key Directories   - `content/articles/`：考情、政策解读类文章   - `content/faq/`：高频问题   - `content/guides/`：备考指南、操作说明   - `app/`：页面路由与入口   - `components/`：通用 UI 组件   - `lib/`：内容解析、数据处理、工具函数   - `scripts/`：搜索索引生成、内容校验脚本   - `generated/`：构建生成的中间数据文件，不要手动修改   - `public/`：静态资源      > 新增代码优先放入已有目录，不随意创建新的顶层目录。      ## Commands      ### Development   - `npm install`：安装依赖   - `npm run dev`：启动本地开发   - `npm run build`：完整构建   - `npm run preview`：预览构建结果      ### Code Quality   - `npm run lint`：代码检查   - `npm run type-check`：类型检查      ### Content   - `npm run content:build`：生成搜索索引和内容数据   - `npm run content:validate`：校验所有内容文件的 frontmatter      > 修改内容文件后，运行 `npm run content:build` 确认索引正常。   > 提交前运行 `npm run lint` 和 `npm run type-check`。      ## Standards      ### 内容规范   - 所有内容文件必须包含完整 frontmatter：title, slug, description, category, tags, date, draft   - 不要修改已发布内容的 slug——slug 变更导致 URL 断裂、搜索失效、内链 404   - 不要随意新增 frontmatter 字段，新字段需确认模板和脚本是否支持   - category 使用已有分类名（政策解读 / 备考指南 / 考情速递 / 常见问题），不发明近义词   - tags 命名保持一致，不混用全称和缩写（如"教资"和"教师资格证"只用一个）   - 文件名使用小写英文 + 中划线，与 slug 保持一致      ### 代码规范   - 优先复用已有组件和工具函数   - 不无理由重命名文件或目录   - 修改公共逻辑前确认影响范围      ### 安全规范   - 不提交 .env 文件、密钥、Token      ## Workflows      ### 新增知识库内容   1. 搜索项目中是否已有类似内容   2. 确认应放入 articles / faq / guides 中的哪个目录   3. 使用统一 frontmatter 结构（参考同目录已有文件）   4. 文件名小写英文 + 中划线，与 slug 保持一致   5. category 和 tags 使用已有名称   6. 完成后运行 `npm run content:build` 验证   7. 检查详情页和列表页渲染是否正常      ### 修改已有内容   1. 先阅读原文件的 frontmatter 和正文结构   2. 确认修改范围：只改正文，还是涉及字段变更   3. 如果修改 category 或 tags，确认聚合页是否受影响   4. 不修改 slug，除非被明确要求且已评估影响   5. 完成后运行 `npm run content:build` 验证      ### 修改内容解析或页面模板   1. 理解数据流：content/ → lib/content.ts → app/ 页面组件   2. 确认改动在哪个环节   3. 小范围修改，先验证单篇内容   4. 完整构建，确认不影响其他内容   5. 检查列表页、详情页、搜索      ## Additional Documentation      处理以下任务前，优先查看对应文档：   - 内容结构与 frontmatter：`docs/content-rules.md`   - 字段与数据实体：`docs/data-schema.md`   - 构建与索引流程：`docs/build-process.md`   - 验证与测试：`docs/testing.md`   - 发布与上线：`docs/deployment.md`      仅在任务相关时读取，不必每次全部加载。      ## Notes   - `generated/` 目录文件由脚本生成，不手动修改   - 若 frontmatter 与规范不一致，记录问题而非大批量修改   - 若文档与代码不一致，以代码为准   - 复杂任务拆解为多个小步骤，不直接大范围改动   ``

# 从模板到你自己的版本：4 步落地

现在你有了模板和示例，但最重要的一步是**把它变成你自己项目的版本**。

不要试图一次写到完美。按这 4 步来：

## 第 1 步：复制最小版模板，填入你的项目信息

花 20 分钟，把模板里的占位符全部替换成你项目的真实信息。重点是：

- 项目一句话描述
    
- 技术栈
    
- 内容目录结构
    
- 常用命令
    
- 3-5 条最关键的 Standards
    

## 第 2 步：用 Claude Code 做 3-5 次真实任务

不要闭门造车。写完初版后，立刻开始用。让 Claude 帮你新增一篇内容、修改一个字段、调整一个页面。

观察它在哪些地方犯错、在哪些地方"不懂规矩"。

## 第 3 步：用 # 快速补充遗漏的规则

每次 Claude 犯了一个"它本不该犯"的错误，就用 `#` 加一条规则：

`# 新增内容时 category 不能自己发明，必须用已有的分类名   `

## 第 4 步：每两周整理一次

积累了一些 `#` 添加的规则后，花 10 分钟回顾一下：

- 有没有重复的？合并
    
- 有没有过时的？删除
    
- 有没有应该移到 docs/ 的？迁移
    
- Standards 是不是超过了 15 条？如果是，考虑精简
    

# 本篇核心认知回顾

**认知一：7 个模块是骨架**

Project Context / About / Key Directories / Commands / Standards / Workflows / Notes——这是 Claude 做事前必须知道的 7 类信息。

**认知二：Standards 只放"做错后代价大"的规则**

不要把 linter 的工作搬到 CLAUDE.md。每条 Standard 都应该指向一个"Claude 不知道就会犯错，而且修复成本很高"的场景。

**认知三：Workflows 防止跳步**

Claude 最大的问题不是不知道规则，而是不知道做事的正确顺序。Workflows 强制它"先看再做"。

**认知四：内容型项目有 5 条专属规则**

frontmatter 必填字段声明、slug 保护、分类标签一致性、新字段约束、修改后验证提醒——这 5 条在普通代码项目中不需要，但在内容项目中必不可少。

**认知五：最好的 CLAUDE.md 是迭代出来的**

先写最小版，用 3-5 次，用 # 补充，定期整理。不要追求一次完美。
