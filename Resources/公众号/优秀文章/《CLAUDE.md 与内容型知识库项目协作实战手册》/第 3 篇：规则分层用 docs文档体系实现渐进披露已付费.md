> 📌 本篇核心目标：学会把"一个膨胀的 CLAUDE.md"拆解为"一个精简的主文件 + 一套专项文档"，掌握 6 份 docs/ 文档各自的职责边界，以及如何让 Claude 按需加载而非全量灌入。

# 当你的 CLAUDE.md 开始变胖

如果你按照第 2 篇的方法写好了 CLAUDE.md，并且真的在项目中用了一段时间，你一定会遇到这个问题：

**它开始膨胀了。**

你用 `#` 补了几条新规则。某次 Claude 改错了构建脚本，你加了一段构建流程说明。分类体系调整过一次，你补了一段详细的分类规则。有人问你"内容文件的 frontmatter 到底有哪些字段"，你把完整的字段清单也塞了进去。

两个月后，你的 CLAUDE.md 从最初的 50 条信息膨胀到了 150 条。

然后你发现一个尴尬的情况：**Claude 开始选择性忽略一些规则了。**

不是因为它"叛逆"，而是因为——回忆第 1 篇讲的"指令预算"概念——当信息量超过阈值后，Claude 对所有规则的遵守力度都会均匀下降。你精心写的 slug 保护规则，和你顺手加的构建步骤说明，在 Claude 眼里没有明显的优先级差异。它们都被淹没在了一大堆文本里。

**这不是 CLAUDE.md 的问题，而是你的信息架构出了问题。**

解决方案不是"删掉一些规则"——那些规则可能确实重要。解决方案是**换一种组织方式**。

# 渐进披露：一个你应该学会的信息架构思路

"渐进披露"（Progressive Disclosure）不是一个新概念。它在产品设计中被广泛使用——

手机设置页面不会一次性把 200 个选项全部展开；它先给你 10 个大类，你点进去才看到子选项。

技术文档站不会把所有 API 堆在首页；它先给你一个导航，你根据当前任务找到对应的文档。

CLAUDE.md 也应该用同样的思路。

## 核心原则

`主文件（CLAUDE.md）        只放核心、高频、长期有效的规则       │       │  "处理 XX 任务前，请查看 docs/XX.md"       │       ▼   专项文档（docs/*.md）       放某一类任务的完整规则和细节   `

CLAUDE.md 是入口，不是仓库。

它的作用是：

1. 承载 Claude 每次都需要知道的基础规则
    
2. 告诉 Claude "某类任务的详细规则在哪个文件"
    

当 Claude 遇到一个具体任务——比如"新增一个 frontmatter 字段"——它先从 CLAUDE.md 中知道"内容结构相关的规则在 `docs/content-rules.md`"，然后读取那份文档，获得完整的指导。

这样做的好处：

**主文件保持精简。** 信号密度高，Claude 不会因为信息过载而"选择性失忆"。

**专项文档可以写得很详细。** 因为它只在相关任务时被加载，不会占用无关任务的上下文空间。

**不同任务获得不同深度的指导。** 改一行正文和改整个数据结构，需要的规则量完全不同。

**维护更容易。** 内容规范变了，你只改 `docs/content-rules.md`，不用在 CLAUDE.md 里大段修改。

## 内容型知识库项目需要哪 6 份文档

不是每个项目都需要同一套文档。但对于内容型知识库项目，以下 6 份文档覆盖了最高频的协作场景：

`docs/   ├── content-rules.md      ← 内容文件怎么写、怎么命名、字段怎么用   ├── data-schema.md        ← 有哪些数据实体、字段含义、实体关系   ├── build-process.md      ← 内容怎么变成页面、构建链路是什么   ├── testing.md            ← 改完东西后怎么验证   ├── code-conventions.md   ← 项目特有的代码约定   └── deployment.md         ← 怎么发布、发布前后检查什么   `

下面逐一拆解每份文档的职责、应该写什么、不该写什么、以及它和 CLAUDE.md 的关系。

# 第 1 份：docs/content-rules.md

## 这份文档的核心使命

**回答一个问题：在这个项目里，一篇"合格的"内容文件应该长什么样？**

它管的是内容的"形式规范"——不是内容写什么（那是编辑的事），而是内容的文件结构、命名方式、字段规则、分类标签使用规则。

## 应该覆盖的核心内容

**1. 文件命名规范**

``## File Naming Rules   - 文件名使用小写英文，单词间用中划线 `-`   - 文件名应反映内容主题，清晰可读   - 不使用空格、中文文件名或无意义缩写   - slug 应与文件名一致或直接映射   ``

这几条看起来简单，但它们防住了一个常见问题：Claude 新增内容时用了不符合项目规范的文件名，导致后续排序、路由或检索出问题。

**2. Frontmatter 完整定义**

这是这份文档最重要的部分。在 CLAUDE.md 里你只写了一句"必须包含 title, slug, category, tags, date, draft"。但在 content-rules.md 里，你需要展开每个字段的含义和使用规则：

``## Frontmatter Schema      ### 基础字段（必填）   - `title`: 页面标题，用于详情页标题和列表展示   - `description`: 内容摘要，用于列表卡片和 SEO 描述   - `slug`: 唯一访问标识，决定页面 URL   - `category`: 主分类，用于导航和聚合页   - `tags`: 标签数组，用于检索和相关推荐   - `date`: 发布时间   - `draft`: 是否为草稿（true 时不进入正式发布流）      ### 扩展字段（可选）   - `updated`: 更新时间   - `featured`: 是否推荐内容   - `order`: 排序权重   - `related_articles`: 相关内容 slug 数组      ### 字段使用规则   - 扩展字段使用前需确认页面模板和构建脚本是否支持   - 不要为单篇内容发明新字段   - 新增字段前，先检查是否有现有字段可以复用   ``

**3. 分类和标签规则**

`## Category Rules   - category 表示主分类，每篇内容只属于一个 category   - 使用已有分类名称，当前项目分类包括：     - 政策解读     - 备考指南     - 考情速递     - 常见问题   - 新分类上线前需确认是否有对应聚合页和导航入口      ## Tag Rules   - tags 用于检索和横向聚合   - 标签简洁明确，不超过 5 个   - 同义词只保留一个写法（如"教资"和"教师资格证"统一用"教师资格证"）   `

把分类名直接列出来，是这份文档最有价值的部分之一。Claude 看到明确的清单，就不会自己发明新分类名。

**4. Slug 变更规则**

`## Slug Rules   - slug 必须唯一   - 已发布内容的 slug 不可修改   - 若必须修改，需同步检查：页面路由、列表聚合、搜索索引、内链引用、sitemap   `

**5. 不同内容类型的模板建议**

`## Content Types      ### Articles（文章）   适合：政策解读、考情分析、深度讲解   推荐正文结构：背景引入 → 核心内容 → 注意事项 → 总结      ### FAQ（常见问题）   适合：高频短问答   推荐结构：问题 → 直接回答 → 补充说明      ### Guides（指南）   适合：操作流程、备考路径   推荐结构：目标说明 → 分步骤讲解 → 注意事项   `

## 这份文档和 CLAUDE.md 的关系

CLAUDE.md 里关于内容的规则，只保留最核心的几条（frontmatter 必填、slug 不能改、分类一致性）。其他所有关于"内容文件怎么写"的细节，全部移到 content-rules.md。

Claude 在处理内容相关任务时，先从 CLAUDE.md 知道基本约束，再从 content-rules.md 获取完整指导。

# 第 2 份：docs/data-schema.md

## 这份文档的核心使命

**回答一个问题：这个项目的数据由哪些实体组成，实体之间是什么关系，字段改动影响什么？**

它管的不是"内容怎么写"（那是 content-rules.md），而是"数据结构怎么设计"。

## 为什么内容项目需要 schema 文档

你可能会觉得："我的项目就是一堆 Markdown 文件，哪来的 schema？"

其实有的。你的 Markdown 文件通过 frontmatter 被解析成结构化数据对象，这些数据对象有字段、有类型、有关系。它就是你的 schema——只不过不是用 SQL 定义的，而是用 frontmatter 和解析函数隐式定义的。

问题是，这种"隐式 schema"Claude 看不到。它不知道 Article 和 Category 的关系是什么，不知道 SearchDocument 是从哪些字段生成的，不知道改一个字段会不会影响搜索索引。

data-schema.md 的作用就是把这些隐式关系**写成显式文档**。

## 应该覆盖的核心内容

**1. 核心实体定义**

列出项目中所有的数据实体，每个实体标注字段和类型：

``## Entity: Article   - `title`: string — 页面标题   - `description`: string — 摘要   - `slug`: string — 唯一标识   - `category`: string — 主分类   - `tags`: string[] — 标签数组   - `date`: string — 发布时间   - `draft`: boolean — 是否草稿   - `content`: string — 正文内容   ``

**2. 实体关系**

`## Entity Relationships   - 一篇 Article 属于一个 Category   - 一篇 Article 可有多个 Tag   - 一个 Topic 可聚合多篇 Article   - SearchDocument 由 Article / FAQ / Guide 转换生成   `

别小看这几句话。Claude 知道了"SearchDocument 是从 Article 转换生成的"，就能理解：改 Article 的字段可能影响搜索索引。没有这句话，它根本想不到。

**3. 高风险变更清单**

`## High-risk Changes      以下变更属于高风险结构变更：   - 修改 slug 规则   - 修改 category 字段语义   - 修改 tags 数据结构（如从 string[] 改为 object[]）   - 删除被多个页面或脚本依赖的字段   - 更改草稿过滤逻辑   `

**4. 变更检查流程**

`## Schema Change Workflow      修改或新增字段时，必须检查：   1. 内容源文件是否需要更新   2. 类型定义是否需要同步   3. 解析函数是否受影响   4. 列表页和详情页是否依赖该字段   5. 搜索索引是否需要收录该字段   6. 是否需要兼容旧数据   `

## 这份文档和 content-rules.md 的边界

容易混淆的点：content-rules.md 和 data-schema.md 有什么区别？

简单说：

- **content-rules.md** 面向"写内容的人"：文件怎么命名、frontmatter 怎么填、分类标签怎么用
    
- **data-schema.md** 面向"改结构的人"：实体有哪些、字段什么含义、改了影响什么
    

如果 Claude 要新增一篇文章，它读 content-rules.md。

如果 Claude 要新增一个 frontmatter 字段或修改数据处理逻辑，它读 data-schema.md。

# 第 3 份：docs/build-process.md

## 这份文档的核心使命

**回答一个问题：内容文件是怎么一步步变成用户看到的页面的？**

## 为什么这份文档对内容项目特别重要

在普通前端项目中，"源码怎么变成页面"这个问题的答案很简单——webpack 或 vite 编译一下就完了。

但在内容型知识库项目中，构建链路要复杂得多：

`content/ 目录下的 Markdown 文件            │            ▼     ┌─────────────┐     │  内容发现     │  扫描目录，找到所有 .md 文件     └──────┬──────┘            ▼     ┌─────────────┐     │  解析         │  提取 frontmatter + 正文     └──────┬──────┘            ▼     ┌─────────────┐     │  验证         │  检查字段完整性、slug 唯一性     └──────┬──────┘            ▼     ┌─────────────┐     │  聚合         │  生成分类数据、标签数据、列表数据     └──────┬──────┘            ▼     ┌─────────────┐     │  索引         │  构建搜索索引     └──────┬──────┘            ▼     ┌─────────────┐     │  渲染         │  页面模板消费数据，生成最终页面     └─────────────┘   `

Claude 不知道这个链路。当你让它"修一下内容解析的 bug"，它可能只看解析函数，不知道这个 bug 会不会影响后面的聚合和索引。当你让它"加一个新的内容类型"，它可能只创建了 Markdown 文件，没想到还需要更新解析逻辑、聚合逻辑和页面模板。

build-process.md 的作用就是让 Claude 在脑子里有这条完整的链路。

## 应该覆盖的核心内容

**1. 构建概览**

用简短的文字或列表说清楚构建的几个阶段。不需要写代码级细节，但需要写清楚"输入是什么 → 经过什么处理 → 输出是什么"。

**2. 输入和输出**

明确标注构建的输入（哪些目录的哪些文件）和输出（生成了什么、放在哪里）。

**3. 构建命令**

和 CLAUDE.md 里的 Commands 模块会有部分重复，但这里可以写得更详细——比如命令的执行顺序、依赖关系。

**4. 常见构建失败原因**

这部分非常实用：

`## Common Failure Points   - frontmatter 缺失必填字段   - slug 重复   - 日期格式异常   - category 拼写不一致   - tags 非数组格式   - 页面模板引用了不存在的字段   - 搜索索引生成脚本与内容结构不同步   `

当 Claude 遇到构建失败时，它可以对照这个清单排查，而不是盲目猜测。

# 第 4 份：docs/testing.md

## 这份文档的核心使命

**回答一个问题：改完东西之后，怎么确认没有搞坏什么？**

## 内容型项目的"测试"和代码项目不同

在代码项目中，测试通常意味着单元测试、集成测试、E2E 测试。但在内容型知识库项目中，最常见的验证需求其实是：

- 新增的内容文件，frontmatter 对不对？
    
- 改了某篇内容的分类，分类聚合页还正常吗？
    
- 改了解析逻辑，所有内容还能正确渲染吗？
    
- 构建出来的搜索索引，新内容被收录了吗？
    
- 草稿内容有没有误发布？
    

这些不全是自动化测试能覆盖的。很多需要"手动检查 + 构建验证"的组合。

## 应该覆盖的核心内容

**1. 测试优先级**

`## Testing Priority      ### P0：必须验证   - frontmatter 完整性   - slug 唯一性   - draft 过滤逻辑      ### P1：应该验证   - 列表页和详情页渲染   - 分类和标签聚合      ### P2：建议验证   - 搜索索引收录   - 相关推荐逻辑   `

分优先级的好处是：Claude 做完一个小改动不需要跑完所有验证，只需要根据改动类型做对应优先级的检查。

**2. 不同类型改动的验证清单**

`## 内容变更后   - [ ] 详情页可正常打开   - [ ] 列表页展示正常   - [ ] 标签和分类链接正常   - [ ] 搜索结果可找到内容（如适用）      ## 结构变更后   - [ ] 类型定义通过   - [ ] 构建脚本正常   - [ ] 搜索索引可生成   - [ ] 旧内容未失效   `

这些检查清单是 testing.md 最有直接价值的部分。Claude 做完任务后可以直接对照执行。

**3. 手动验证的标准动作**

内容型项目有很多问题只能通过"打开页面看一看"来发现。testing.md 应该写清楚手动验证时该看哪些页面、检查什么。

# 第 5 份：docs/code-conventions.md

## 这份文档的核心使命

**回答一个问题：在这个项目里写代码，有哪些不是 linter 管得了但必须遵守的约定？**

## 注意：这不是 linter 的替代品

第 2 篇已经讲过，代码格式相关的规则不要写进 CLAUDE.md。同样的道理也适用于这份文档——code-conventions.md 不是用来写"缩进几格、用单引号还是双引号"的。

它应该写的是**项目层面的实现约定**。

## 对内容型项目来说，最重要的约定是什么

**内容解析逻辑的集中管理**

``- frontmatter 解析逻辑应集中在 `lib/content.ts`   - 不要在页面组件中各自重复解析逻辑   - 公共内容转换函数放入 lib/，不散落在各处   ``

**类型定义的集中管理**

``- 内容实体类型集中定义（如 `types/content.ts`）   - 新增字段时先改类型定义，再改解析和页面   - 不允许页面依赖"隐式存在但未定义"的字段   ``

**页面数据消费规则**

`- 页面应消费标准化后的数据对象，不直接依赖原始内容文件细节   - 列表页、详情页、专题页尽量复用同一套字段语义   `

这些约定的共同目标是：**防止内容处理逻辑散落在各处，导致改一处要同步改十处。**

# 第 6 份：docs/deployment.md

## 这份文档的核心使命

**回答一个问题：怎么把内容安全地发布上线，发布后怎么确认没出问题？**

## 内容型项目的发布和纯代码项目有什么不同

纯代码项目的发布，核心关注点是"构建成功 + 功能正常"。

内容型项目的发布，还需要额外关注：

- 新增内容是否出现在正确位置
    
- 草稿内容是否被误发布
    
- 分类和标签聚合是否正常
    
- 搜索索引是否包含新内容
    
- slug 是否有冲突
    

这些都不是"构建通过"就能保证的。

## 应该覆盖的核心内容

**1. 标准发布流程**

写清楚从"准备发布"到"确认完成"的完整步骤。

**2. 发布前检查清单 + 发布后检查清单**

这是 deployment.md 最有直接价值的部分：

`## Pre-release Checklist   - [ ] frontmatter 合法   - [ ] slug 无冲突   - [ ] 草稿不会误发布   - [ ] 构建通过   - [ ] 环境变量齐全      ## Post-release Checklist   - [ ] 新内容详情页正常   - [ ] 列表页收录正常   - [ ] 分类和标签聚合正常   - [ ] 搜索结果正常   - [ ] 无 404 或空页   `

**3. 回滚策略**

简要说明出问题后怎么回滚，以及常见的问题原因排查方向。

# 6 份文档之间的职责边界

把 6 份文档放在一起看，它们各自的边界是这样的：

`你要新增一篇内容     → 读 content-rules.md（怎么写、怎么命名、字段怎么填）      你要新增一个 frontmatter 字段     → 读 data-schema.md（字段含义、影响哪些环节、变更流程）      构建失败了     → 读 build-process.md（构建链路、常见失败原因）      改完东西想确认没搞坏什么     → 读 testing.md（验证清单、手动检查点）      要改内容解析逻辑或页面模板     → 读 code-conventions.md（代码组织规则、数据消费约定）      准备发布上线     → 读 deployment.md（发布流程、检查清单、回滚策略）   `

它们之间有少量交叉，但大方向上不重叠。如果你发现某段内容不知道该放哪份文档，用这个判断标准：**这段内容是在什么场景下被需要的？** 谁需要它，它就放在谁的文档里。

# 如何在 CLAUDE.md 中引用这些文档

有了 6 份 docs/，你需要在 CLAUDE.md 里告诉 Claude 它们的存在。

## 推荐写法

在 CLAUDE.md 末尾（或 Notes 之前）加一个模块：

``## Additional Documentation      处理以下任务前，请优先查看对应文档：   - 内容结构与 frontmatter：`docs/content-rules.md`   - 字段与数据实体：`docs/data-schema.md`   - 构建与索引流程：`docs/build-process.md`   - 验证与测试：`docs/testing.md`   - 项目实现约定：`docs/code-conventions.md`   - 发布与上线：`docs/deployment.md`      仅在任务相关时读取对应文档，不必每次全部加载。   ``

## 最后一句话为什么重要

"仅在任务相关时读取对应文档，不必每次全部加载。"

这句话不是客套话，而是一条**功能性指令**。

如果你不写这句话，Claude 有可能在每次会话开始时把 6 份文档全部读一遍——这会消耗大量上下文空间，而且大部分内容和当前任务无关。

加了这句话，Claude 的行为模式变成：

1. 先看 CLAUDE.md，获得项目基础认知
    
2. 分析当前任务属于什么类型
    
3. 只读取和当前任务相关的那份（或那几份）docs/
    
4. 开始工作
    

这就是"渐进披露"的实际效果——信息按需加载，不浪费上下文预算。

## 引用粒度怎么控制

有些人会写得更细：

``# ❌ 过于细碎的引用   - 新增文章时先读 `docs/content-rules.md` 的"File Naming Rules"部分   - 修改 frontmatter 时先读 `docs/content-rules.md` 的"Frontmatter Schema"部分   - 修改 slug 时先读 `docs/content-rules.md` 的"Slug Rules"部分   - ……列出了 20 条精确引用   ``

不需要这么细。Claude 足够聪明，你告诉它"内容相关看 content-rules.md"，它会自己找到相关段落。过于细碎的引用反而增加了 CLAUDE.md 的信息量。

按**任务类型**指向就够了，不需要按**文档段落**指向。

# 索引文件：大型项目的补充工具

如果你的项目代码量较大、模块较多、文件较分散，可以额外维护一两份索引文件，帮 Claude 快速了解代码地图。

## general_index.md

描述每个目录和关键文件的作用：

``# General Index      ## /lib/   - `content.ts`：内容文件解析、frontmatter 提取   - `search.ts`：搜索功能实现   - `utils.ts`：通用工具函数      ## /scripts/   - `build-search-index.ts`：搜索索引生成脚本   - `validate-content.ts`：内容文件校验脚本      ## /app/   - `page.tsx`：首页   - `[category]/page.tsx`：分类聚合页   - `[slug]/page.tsx`：内容详情页   ``

## 一个关键提醒

索引文件必须加一句声明：

`> 这些索引仅供导航参考，可能不是最新的。必要时请查看实际代码确认。   `

为什么？因为索引文件很容易过期。你加了一个新文件但忘了更新索引，Claude 如果完全信赖索引，可能会忽略那个新文件。加了这句话，Claude 知道"索引是参考，不是绝对真相"，遇到不确定的情况会去看实际代码。

## 索引文件的引用方式

在 CLAUDE.md 里加一句：

``## Navigation      项目提供了索引文件辅助代码导航：   - `general_index.md`：目录和文件说明      索引可能不是最新的，遇到不一致时以实际代码为准。   ``

## 完整的项目结构一览

把所有文件放在一起，你的项目结构应该是这样的：

`project/   ├── CLAUDE.md                    ← 核心规则入口   ├── CLAUDE.local.md              ← 个人偏好（不提交 Git）   ├── general_index.md             ← 代码导航（可选）   ├── docs/   │   ├── content-rules.md         ← 内容规范   │   ├── data-schema.md           ← 数据结构   │   ├── build-process.md         ← 构建流程   │   ├── testing.md               ← 验证策略   │   ├── code-conventions.md      ← 代码约定   │   └── deployment.md            ← 发布流程   ├── content/   │   ├── articles/   │   ├── faq/   │   └── guides/   ├── app/   ├── components/   ├── lib/   ├── scripts/   └── ...   `

上半部分是"规则层"——CLAUDE.md + docs/ + 索引。

下半部分是"代码层"——应用代码和内容文件。

规则层指导代码层的协作方式。代码层的变化反过来驱动规则层的更新。

# 一个实际的工作场景演示

用一个具体场景说明这套分层体系是怎么工作的。

**场景：你让 Claude 给知识库新增一个"专题"内容类型**

这是一个偏复杂的任务，涉及内容、数据结构、构建、页面多个层面。

**Claude 的处理过程（有了分层体系后）：**

第 1 步：读 CLAUDE.md，了解项目基础信息。知道这是内容型站点，内容在 content/ 目录下，用 frontmatter 定义元数据。

第 2 步：判断任务涉及"新增内容类型"，这同时关系到内容规范和数据结构。按 Additional Documentation 的指引，读取 `docs/content-rules.md` 和 `docs/data-schema.md`。

第 3 步：从 content-rules.md 了解到现有的内容类型（articles、faq、guides）和 frontmatter 规范。知道新增内容类型需要保持一致的字段结构。

第 4 步：从 data-schema.md 了解到各实体的关系，知道 Topic 可以聚合多篇 Article。了解到新增实体需要检查类型定义、解析函数、页面模板和搜索索引。

第 5 步：从 CLAUDE.md 的 Workflows 中找到"修改内容解析或页面模板"的步骤，按步骤执行。

第 6 步：完成后，按 testing.md 中"结构变更后"的检查清单验证。

**如果没有这套分层体系：**

Claude 大概率会直接创建一个 `content/topics/` 目录，往里面扔一个 Markdown 文件，然后告诉你"完成了"。至于类型定义、解析函数、聚合逻辑、搜索索引、页面路由——全部遗漏。

**差距就在这里。**

# 什么时候该拆、什么时候不拆

并不是所有项目都需要 6 份完整的 docs/。

## 建议拆的场景

- 你的 CLAUDE.md 已经超过 80 条信息，开始感觉"塞不下了"
    
- 你有多种类型的协作任务（内容、代码、构建、发布），每种需要不同深度的规则
    
- 项目有 2 人以上协作，需要共享规范
    
- 项目运行时间超过 3 个月，规则积累了一定量
    

## 可以不拆的场景

- 项目很小、很新，规则还没积累起来
    
- 你的 CLAUDE.md 还控制在 50 条信息以内，Claude 遵守得很好
    
- 你是一个人维护，所有规则你自己都记得住
    

## 建议的拆分顺序

如果你决定拆，不要一次建 6 份。按优先级来：

`第一优先：docs/content-rules.md     → 内容规范是内容型项目的第一基础设施      第二优先：docs/data-schema.md     → 防止 Claude 乱改数据结构      第三优先：docs/build-process.md     → 让 Claude 理解完整的数据链路      第四优先：docs/testing.md     → 建立"改完就验"的习惯      最后再补：docs/code-conventions.md + docs/deployment.md     → 这两份在项目更成熟时价值更大   `

先建最重要的 3 份，用起来。后面按需补充。

# 本篇核心认知回顾

**认知一：CLAUDE.md 是入口，不是仓库**

主文件只放核心、高频、长期有效的规则。专项细节拆到 docs/ 中。

**认知二：渐进披露的核心是"按需加载"**

Claude 不需要每次都读完所有文档。它根据当前任务类型，只读取相关的那份 docs/。这样既保持了主文件的精简，又不丢失规则的深度。

**认知三：6 份文档各有明确的职责边界**

content-rules 管"怎么写内容"、data-schema 管"数据结构怎么设计"、build-process 管"内容怎么变成页面"、testing 管"怎么验证"、code-conventions 管"代码怎么组织"、deployment 管"怎么发布"。

**认知四：引用方式要克制**

在 CLAUDE.md 里按任务类型指向 docs/，加一句"仅在任务相关时读取"。不需要精确到文档的每个段落。

**认知五：索引文件是补充，不是替代**

索引帮助 Claude 快速了解代码地图，但必须标注"可能不是最新的"，防止 Claude 过度依赖过期信息。

# 本篇练习

**练习：为你的项目建立前 3 份 docs/**

1. 创建 `docs/` 目录
    
2. 建立 `docs/content-rules.md`：把你项目的 frontmatter 字段、分类列表、标签规则、文件命名规范写进去
    
3. 建立 `docs/data-schema.md`：列出你项目的核心数据实体和字段
    
4. 建立 `docs/build-process.md`：画出你项目从"内容文件"到"最终页面"的链路
    
5. 在 CLAUDE.md 中加上 Additional Documentation 引用段
    
6. 用 Claude Code 做一次"新增 frontmatter 字段"的任务，观察它是否会去读 docs/
    

完成这个练习后，你就有了一套"主文件 + 3 份专项文档"的基本体系。
