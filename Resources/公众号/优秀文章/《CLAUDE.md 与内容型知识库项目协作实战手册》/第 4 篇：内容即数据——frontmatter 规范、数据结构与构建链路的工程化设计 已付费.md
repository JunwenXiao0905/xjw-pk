# 第一个认知转变：内容文件不是文本

你打开 `content/articles/teacher-cert-guide.md`，看到的是这样的东西：

`---   title: 2026年教师资格证报考完整指南   description: 从报名条件到考试科目，一文搞懂教资报考全流程   slug: teacher-cert-guide   category: 备考指南   tags: [教师资格证, 报考流程, 2026]   date: 2026-01-15   draft: false   ---      # 2026年教师资格证报考完整指南      教师资格证考试是……   `

看起来就是一篇 Markdown 文章，对吧？

**但从系统的角度看，它同时是 8 个东西：**

`这篇 Markdown 文件，同时是：      1. 详情页的数据源       → title、description、content 被渲染成详情页   2. 列表页的一个条目      → title、description、date 出现在文章列表里   3. 分类聚合页的成员      → category="备考指南" 让它出现在"备考指南"分类页   4. 标签聚合页的成员      → tags 让它出现在"教师资格证"等标签页   5. 搜索索引的一条记录    → title、description、tags、content 被写入搜索数据   6. 路由系统的一个映射    → slug="teacher-cert-guide" 决定 URL 是 /teacher-cert-guide   7. SEO 的元数据来源     → title 和 description 影响搜索引擎收录   8. 内链和推荐的被引用对象 → 其他文章的 related_articles 可能引用它的 slug   `

**Claude 默认不知道这些。**

它看到一个 Markdown 文件，只会把它当"一篇文本"来处理。你让它"改个分类"，它就改了——不会想到这个改动还影响分类聚合页。你让它"优化一下 slug"，它就改了——不会想到旧 URL 会 404、搜索引擎已收录的链接会失效。

你的 docs/content-rules.md、docs/data-schema.md 和 docs/build-process.md 的核心任务，就是把这些"隐性连锁关系"变成"Claude 能看到的显性规则"。

下面一个模块一个模块来讲。

# Frontmatter 设计方法论

Frontmatter 是内容文件的"头部元数据"，是整个内容系统的数据基础。它设计得好不好，直接决定了你的内容系统是"越用越稳"还是"越用越乱"。

## 设计原则：稳定性 > 完整性

很多人设计 frontmatter 的思路是"我需要什么字段就加什么字段"。结果三个月后，不同时期新增的内容文件，frontmatter 五花八门——有的有 `summary`，有的没有；有的用 `author`，有的用 `writer`；有的 `tags` 是数组，有的是逗号分隔的字符串。

正确的设计思路是：**先定一套稳定的基础字段，所有内容文件严格遵守。扩展字段只在确认被消费后才启用。**

"稳定"的意思是：

- 字段名不变
    
- 字段类型不变
    
- 字段语义不变
    
- 是否必填不变
    

一旦某个字段进入了"基础字段"清单，就不要轻易改动它。因为改动一个基础字段，意味着你需要同时更新：所有内容文件、类型定义、解析函数、页面模板、搜索索引脚本——任何一个环节遗漏，都会出问题。

## 基础字段：你必须有的 7 个

不管你的项目多大多小，以下 7 个字段基本是内容型知识库项目的标配：

`---   title:""          # 页面标题   description:""    # 内容摘要   slug:""           # 唯一访问标识   category:""       # 主分类   tags:[]           # 标签数组   date:""           # 发布时间   draft:false       # 是否草稿   ---   `

逐个说明为什么这 7 个是必备的：

**title**

最基础的字段，没什么争议。用于详情页标题、列表页展示、SEO 标题、搜索索引。

唯一需要注意的是：title 是"给人看的标题"，不是文件名。文件名用 slug 风格（小写英文+中划线），title 用自然语言。

**description**

内容摘要。用于列表页卡片、SEO meta description、搜索结果的展示片段。

很多人在早期会省略这个字段，结果列表页的摘要显示要么是空的，要么是用正文前 100 字截取的——效果很差。

建议：description 设为必填，长度控制在 50-150 字之间。

**slug**

整个 frontmatter 中最关键的字段，没有之一。它决定了页面的 URL，是内容在整个系统中的"身份证"。

后面会专门用一整节来讲 slug 规则。

**category**

主分类。每篇内容只属于一个 category。用于导航、分类聚合页、侧边栏分组。

category 和 tags 的区别后面会详细讲。

**tags**

标签数组。用于横向关联、搜索过滤、相关推荐。

注意 tags 必须是数组格式（`tags: [标签A, 标签B]`），不要用逗号分隔的字符串。统一格式能避免解析问题。

**date**

发布时间。用于排序、时间线展示、"最新内容"功能。

格式建议统一为 `YYYY-MM-DD`。不要有的文件用 `2026-01-15`，有的用 `2026/01/15`，有的用 `Jan 15, 2026`。格式不一致会导致排序出错。

**draft**

是否为草稿。`draft: true` 的内容不应该出现在正式的列表页、搜索索引和聚合页中。

这个字段看起来简单，但它的影响范围很大——构建脚本、搜索索引、列表查询、RSS 生成，所有环节都需要正确过滤草稿。一旦过滤逻辑有漏洞，草稿内容就会被公开发布。

## 扩展字段：怎么判断该不该加

基础字段之外，你可能还需要一些扩展字段：

`updated: ""            # 更新时间   featured: false        # 是否推荐   order: 0               # 排序权重   summary: ""            # 更短的摘要（用于卡片等小空间）   related_articles: []   # 相关内容 slug 列表   source: ""             # 内容来源   author: ""             # 作者   `

但不是所有字段都应该加。加一个字段的隐性成本包括：

- 类型定义要更新
    
- 解析函数可能要调整
    
- 页面模板如果不消费这个字段，它就是"死数据"
    
- 未来所有新增内容都要考虑是否填这个字段
    
- Claude 新增内容时要知道这个字段的规则
    

**判断标准：这个字段有没有"消费者"？**

如果某个字段被页面模板渲染、被搜索索引收录、被聚合逻辑使用——那它有消费者，值得加。

如果某个字段"感觉将来可能有用"但现在没有任何代码依赖它——不要加。等真正需要时再加。

这不是偷懒，而是控制复杂度。每个字段都是一个维护负担，空字段尤其如此——Claude 看到一个字段却不知道它有什么用、该不该填，反而增加了出错概率。

## 在 content-rules.md 中怎么写

把上面的分析转化成文档：

`## Frontmatter Schema      ### 基础字段（必填）      | 字段 | 类型 | 说明 |   |------|------|------|   | title | string | 页面标题 |   | description | string | 内容摘要，50-150 字 |   | slug | string | 唯一访问标识，决定 URL |   | category | string | 主分类，必须使用已有分类名 |   | tags | string[] | 标签数组 |   | date | string | 发布时间，格式 YYYY-MM-DD |   | draft | boolean | 是否草稿 |      ### 扩展字段（可选，需确认消费方存在）      | 字段 | 类型 | 说明 |   |------|------|------|   | updated | string | 更新时间 |   | featured | boolean | 是否推荐内容 |   | order | number | 排序权重 |      ### 字段使用规则   - 基础字段全部必填，不允许缺失   - 扩展字段使用前需确认有对应的页面消费或索引收录   - 不要为单篇内容发明新字段   - 新增字段需同步更新：类型定义、解析函数、相关页面模板   `

用表格而不是纯文本列表，是因为 Claude 对表格的结构化信息解读更准确。字段名、类型、说明三列对齐，Claude 一眼就能看清每个字段是什么。

# Slug 规则：frontmatter 中优先级最高的字段

前面说了，slug 是最关键的字段。这里用一整节来讲清楚。

## 为什么 slug 这么特殊

因为 slug 是**连接内容和系统的唯一标识**。

`slug = "teacher-cert-guide"       │       ├── URL：/teacher-cert-guide       ├── 路由匹配：app/[slug]/page.tsx 用它找到对应内容       ├── 搜索索引：索引中用 slug 作为文档 ID       ├── 内链引用：其他文章的 related_articles 引用这个 slug       ├── SEO：搜索引擎用 URL 建立索引       └── 外部链接：用户可能已经收藏或分享了这个 URL   `

改一个 slug，上面 6 个环节全部受影响。

这就是为什么 CLAUDE.md 和 content-rules.md 都要反复强调"不要改 slug"——不是规则洁癖，而是改动成本太高。

## Slug 命名规范

``## Slug Rules      ### 命名规范   - 使用小写英文   - 单词间用中划线 `-`   - 简洁但有描述性   - 与文件名保持一致或直接映射      ### 好的示例   - `teacher-cert-guide`   - `gongkao-interview-faq`   - `react-form-validation`      ### 不好的示例   - `article-001`（无描述性）   - `Teacher_Cert_Guide`（大写+下划线）   - `2026年教资报考指南`（中文）   - `temp-draft-final-v2`（临时命名）   ``

## Slug 变更的完整检查流程

如果确实需要修改一个已发布内容的 slug（极少数情况），content-rules.md 应该写清楚完整的检查流程：

`### Slug 变更检查流程      如果必须修改已发布内容的 slug：   1. 确认变更原因，评估是否真的必要   2. 搜索项目中所有引用旧 slug 的位置：      - 其他内容文件的 related_articles      - 页面组件中的硬编码链接      - 导航或菜单配置   3. 检查搜索索引中是否有旧 slug 的记录   4. 确认是否需要设置 URL 重定向（旧 URL → 新 URL）   5. 同步更新所有引用   6. 重新构建并验证所有关联页面   `

这个流程不是给你看的——它是给 Claude 看的。当 Claude 被要求修改 slug 时，它会按这个流程走，而不是直接改文件了事。

# 分类与标签治理

## category 和 tags 的本质区别

很多项目把 category 和 tags 混为一谈。其实它们有本质区别：

`category（分类）   ├── 结构性的：决定内容在导航、聚合页中的位置   ├── 唯一的：每篇内容只属于一个分类   ├── 稳定的：分类体系不轻易调整   └── 影响大的：改分类等于改内容在站点中的"归属"      tags（标签）   ├── 检索性的：服务于搜索和横向关联   ├── 多值的：每篇内容可以有多个标签   ├── 灵活的：可以较自由地新增标签   └── 影响小的：改标签主要影响搜索和推荐   `

在 data-schema.md 中应该写清楚这个区别。

## 分类治理的核心问题：防止"近义分类"

Claude 在内容项目中最隐蔽的错误之一，就是创建近义分类。

你的项目有一个分类叫"政策解读"。Claude 新增一篇内容时用了"政策分析"。语义上很接近，但系统层面完全是两个东西——"政策分析"没有对应的聚合页，这篇文章从分类导航中"消失"了。

**最有效的防范方式：在 content-rules.md 中明确列出所有合法分类名。**

`## Category Rules      当前项目的合法分类：      | 分类名 | slug | 说明 |   |--------|------|------|   | 政策解读 | policy | 政策文件解读和变化分析 |   | 备考指南 | study-guide | 备考方法、学习路径 |   | 考情速递 | exam-news | 考试动态、时间节点 |   | 常见问题 | faq | 高频问答 |   - 新增内容必须使用上表中的分类名，不允许发明新分类   - 若确需新增分类，需同步创建对应聚合页和导航入口   `

把分类做成表格并写上 slug，Claude 就不可能搞错了。它要给一篇文章选分类，直接从表里挑，而不是自己编。

### 标签治理的核心问题：防止"同义标签"

标签的问题和分类类似，但更容易失控。因为标签是多值的、比较自由的，Claude 特别容易创造同义标签：

- "教资" vs "教师资格证" vs "教师资格"
    
- "React" vs "react" vs "React.js"
    
- "面试" vs "面试技巧" vs "面试准备"
    

**两个治理策略：**

策略一：维护一份标签白名单（适合标签量不大的项目）

`## Tag Rules      常用标签清单：      教师资格证、教师编制、特岗教师、报名流程、   考试科目、面试技巧、备考计划、政策变化、   2026、各省考情   - 优先使用已有标签   - 同义词只保留一个写法   - 新标签需有 3 篇以上内容才值得创建   `

策略二：定义命名规范（适合标签量大的项目）

`## Tag Naming Rules   - 使用全称，不用缩写（"教师资格证"而非"教资"）   - 不加冗余后缀（"面试"而非"面试技巧"、"面试准备"）   - 专有名词保持大小写一致   - 新增标签前搜索已有标签是否有同义词   `

# Schema 思维：把隐式结构变成显式文档

## 什么是内容项目的 schema

你的项目没有数据库，但它有 schema。

这个 schema 由三部分组成：

1. **Frontmatter 定义的字段**——就是上面讲的 title、slug、category 等
    
2. **解析函数生成的数据对象**——Markdown 文件经过解析后，变成了 `{ title, slug, content, ... }` 这样的 JavaScript 对象
    
3. **实体之间的关系**——Article 属于 Category，Article 有多个 Tag，Topic 聚合多个 Article
    

在代码项目中，schema 通常由数据库的表定义或 ORM 模型来管理，有明确的来源和约束。

但在内容型项目中，schema 是"散装的"——一部分在 frontmatter 里，一部分在解析函数里，一部分在类型定义里，一部分只存在于开发者的脑子里。

Claude 无法从代码中自动拼出完整的 schema。你需要在 data-schema.md 中把它写成一份显式的文档。

## 核心实体设计

内容型知识库项目通常有以下核心实体：

**Article（文章）**

最基础的内容实体。字段就是 frontmatter 的 7 个基础字段 + content。

**Guide（指南）**

和 Article 结构类似，但可能多几个字段：

``## Entity: Guide      继承 Article 基础字段，额外可用：   - `order`: number — 排序权重（指南通常有阅读顺序）   - `prerequisites`: string[] — 前置阅读的 slug 列表   ``

**FAQItem（常见问题条目）**

结构更简单：

``## Entity: FAQItem   - `question`: string — 问题   - `answer`: string — 回答（即 content）   - `slug`: string   - `category`: string   - `tags`: string[]   - `date`: string   - `draft`: boolean   ``

**Topic（专题）**

用于聚合多篇相关内容：

``## Entity: Topic   - `title`: string   - `description`: string   - `slug`: string   - `related_articles`: string[] — 聚合的文章 slug 列表   - `order`: number   - `draft`: boolean   ``

**Category 和 Tag**

这两个实体可能不是以 Markdown 文件的形式存在，而是从所有内容文件的 frontmatter 中自动提取的。但它们依然是你系统中的实体：

``## Entity: Category   - `name`: string — 分类名   - `slug`: string — 分类标识   - 来源：从所有内容文件的 category 字段聚合      ## Entity: Tag   - `name`: string — 标签名   - `slug`: string — 标签标识   - 来源：从所有内容文件的 tags 字段聚合   ``

**SearchDocument（搜索文档）**

这是一个派生实体——不是直接由内容文件定义的，而是在构建时从其他实体转换生成的：

``## Entity: SearchDocument   - `id`: string — 通常是 slug   - `type`: string — 来源类型（article / faq / guide）   - `title`: string   - `description`: string   - `slug`: string   - `category`: string   - `tags`: string[]   - `content`: string — 用于全文搜索的正文内容   - 来源：由 Article / Guide / FAQItem 在构建时转换生成   ``

## 实体关系图

在 data-schema.md 中画一张关系图，让 Claude 一眼看清楚：

`┌──────────┐     belongs to     ┌──────────┐   │ Article  │ ──────────────── → │ Category │   │ Guide    │                    └──────────┘   │ FAQItem  │   └─────┬────┘         │ has many         ▼   ┌──────────┐   │   Tag    │   └──────────┘      ┌──────────┐     aggregates     ┌──────────┐   │  Topic   │ ──────────────── → │ Article  │   └──────────┘                    └──────────┘      ┌──────────┐     generated from ┌──────────┐   │ Search   │ ← ─────────────── │ Article  │   │ Document │                    │ Guide    │   └──────────┘                    │ FAQItem  │                                   └──────────┘   `

这张图让 Claude 理解：修改 Article 的字段，可能影响 SearchDocument；修改 Category 的命名，会影响所有属于该分类的 Article 的聚合结果。

## 高风险结构变更清单

在 data-schema.md 中明确标注哪些变更是"高风险"的：

`## High-risk Changes      以下变更影响范围大，需要格外谨慎：   1. **修改 slug 规则**      影响：URL、路由、搜索索引、内链、SEO      2. **修改 category 字段语义或名称**      影响：聚合页、导航、所有该分类下的内容   3. **修改 tags 的数据结构**      影响：标签聚合、搜索过滤、相关推荐      示例风险：从 string[] 改为 object[] 会破坏所有现有解析逻辑   4. **删除被依赖的字段**      影响：所有消费该字段的页面模板、解析函数、搜索索引   5. **修改 draft 过滤逻辑**      影响：可能导致草稿内容被公开发布   6. **修改 SearchDocument 的生成逻辑**      影响：搜索结果的完整性和准确性   `

每个风险项都标注了"影响什么"，这样 Claude 在遇到相关任务时会自动警觉。

## Schema 变更的标准检查流程

`## Schema Change Workflow      修改数据结构时，按以下顺序检查：   1. 明确变更目标和原因   2. 检查类型定义（如 types/content.ts）   3. 检查解析函数（如 lib/content.ts）   4. 检查页面消费逻辑（详情页、列表页、聚合页）   5. 检查搜索索引生成脚本   6. 检查旧数据的兼容性   7. 更新 docs/data-schema.md   8. 运行完整构建验证   `

这个流程不是走形式——它确保每次结构变更都经过完整的影响评估。

# 构建链路：从 content/ 到页面的完整数据流

## 为什么 Claude 需要理解构建链路

当 Claude 遇到一个问题——比如"某篇文章在搜索结果里找不到"——它需要知道内容是怎么一步步进入搜索索引的，才能准确定位问题在哪个环节。

如果它不理解构建链路，只会盲目猜测："可能是内容没写对？可能是搜索功能坏了？"

如果它理解了链路，它会有条理地排查："内容文件有没有？frontmatter 完整吗？draft 是不是 true？解析函数有没有正确处理？索引生成脚本有没有收录这条记录？"

## 一个典型的内容站构建链路

`Stage 1：内容发现   ┌─────────────────────────────────────┐   │ 扫描 content/ 目录                   │   │ 找到所有 .md / .mdx 文件              │   │ 按子目录判断内容类型                    │   │   content/articles/ → type: article  │   │   content/faq/      → type: faq      │   │   content/guides/   → type: guide    │   └──────────────┬──────────────────────┘                  ▼   Stage 2：解析   ┌─────────────────────────────────────┐   │ 读取文件内容                          │   │ 提取 frontmatter（gray-matter 等库）  │   │ 分离 frontmatter 和正文               │   │ 生成标准化数据对象                     │   └──────────────┬──────────────────────┘                  ▼   Stage 3：验证   ┌─────────────────────────────────────┐   │ 检查 frontmatter 必填字段             │   │ 检查 slug 唯一性                     │   │ 检查 category 是否合法               │   │ 检查 date 格式                       │   │ 过滤 draft: true 的内容              │   └──────────────┬──────────────────────┘                  ▼   Stage 4：聚合   ┌─────────────────────────────────────┐   │ 按 category 分组 → 分类数据           │   │ 提取所有 tags → 标签数据              │   │ 按 date 排序 → 时间线数据             │   │ 按 Topic 配置聚合 → 专题数据          │   │ 计算 related → 相关推荐数据           │   └──────────────┬──────────────────────┘                  ▼   Stage 5：索引   ┌─────────────────────────────────────┐   │ 从标准化对象生成 SearchDocument        │   │ 写入搜索索引文件                      │   │ （可能还有 sitemap、RSS 等）           │   └──────────────┬──────────────────────┘                  ▼   Stage 6：渲染   ┌─────────────────────────────────────┐   │ 页面模板消费标准化数据                 │   │ 详情页：渲染单篇内容                  │   │ 列表页：渲染内容列表                  │   │ 分类页：渲染某分类下的内容             │   │ 标签页：渲染某标签下的内容             │   │ 搜索页：基于搜索索引提供搜索功能        │   └─────────────────────────────────────┘   `

## 在 build-process.md 中怎么写

不需要把上面这张图完整复制进去（那太长了）。你需要写的是：

**1. 构建的核心阶段（简要描述）**

让 Claude 知道构建分几步，每步做什么。

**2. 输入和输出**

``## Inputs   - `content/`：Markdown 内容文件   - `lib/content.ts`：解析逻辑   - `scripts/build-search-index.ts`：索引生成脚本      ## Outputs   - 页面数据（被 app/ 中的页面模板消费）   - `generated/search-index.json`：搜索索引   ``

**3. 常见构建失败原因**

这是最实用的部分：

`## Common Failure Points   - frontmatter 缺失 title 或 slug   - slug 重复（两篇内容用了同一个 slug）   - date 格式不一致（如混用 YYYY-MM-DD 和 YYYY/MM/DD）   - category 拼写不一致（如"备考指南"和"备考攻略"）   - tags 格式错误（如写成字符串而非数组）   - draft 过滤逻辑不生效（draft: true 的内容出现在正式列表中）   - 搜索索引脚本未同步更新（新增字段后索引脚本未收录）   - 页面模板引用了不存在的 frontmatter 字段   `

**4. 构建变更检查流程**

`## Build Change Workflow      修改构建相关逻辑时：   1. 确认是内容问题、解析问题还是渲染问题   2. 理解当前输入数据和输出结果   3. 小范围修改，先用单篇内容验证   4. 完整构建，确认不影响其他内容   5. 检查列表页、详情页、搜索   `

# 把三份文档串起来看

到这里，你已经理解了 content-rules.md、data-schema.md 和 build-process.md 分别应该写什么。

把它们放在一起，它们的协作关系是这样的：

`Claude 要新增一篇文章     │     ├── 读 content-rules.md     │     知道：文件怎么命名、frontmatter 怎么填、分类用哪个、标签怎么选     │     └── 写完之后运行构建           如果构建失败 → 读 build-process.md 排查           如果涉及新字段 → 读 data-schema.md 评估影响      Claude 要新增一个 frontmatter 字段     │     ├── 读 data-schema.md     │     知道：现有实体有哪些字段、新字段影响什么、变更流程是什么     │     ├── 读 build-process.md     │     知道：新字段是否需要被搜索索引收录、是否影响聚合逻辑     │     └── 改完后更新 content-rules.md           把新字段加入 frontmatter schema      Claude 排查"某篇文章搜索不到"     │     ├── 读 build-process.md     │     按链路排查：内容文件有没有 → frontmatter 对不对 → draft 是否 false     │     → 解析有没有问题 → 搜索索引有没有收录     │     └── 如果是字段问题 → 读 data-schema.md         如果是内容格式问题 → 读 content-rules.md   `

三份文档不是独立的——它们形成一个相互支撑的知识网络。

# 一个容易被忽视的问题：草稿规则的系统级影响

最后重点讲一个很多人低估的问题：`draft` 字段。

`draft: true` 看起来就是一个布尔值，但它的影响是系统级的：

`draft: true 的内容不应该出现在：      - 正式列表页   - 分类聚合页   - 标签聚合页   - 搜索索引   - RSS 输出   - Sitemap   - "最新内容"模块   - "相关推荐"结果中   `

如果上面任何一个环节的过滤逻辑有漏洞，草稿就会被公开发布。

**在 data-schema.md 中应该明确标注：**

``## Draft Rules   - `draft: true` 的内容默认不进入正式发布流   - 构建时所有内容查询逻辑必须显式过滤 draft   - 搜索索引生成时必须排除 draft 内容   - 预览环境可展示 draft 内容，但正式环境不允许   - 修改 draft 过滤逻辑属于高风险变更   ``

这几条规则看起来啰嗦，但它们防住的是一个真实的、严重的、而且不容易被发现的问题。

# 本篇核心认知回顾

**认知一：内容文件不是文本，而是系统输入**

一篇 Markdown 同时是 8 个系统环节的数据源。Claude 不会自动意识到这些连锁关系，你需要通过文档显式告知。

**认知二：frontmatter 设计追求稳定性，不追求完整性**

先定 7 个必填基础字段，严格遵守。扩展字段只在有消费者时才启用。字段稳定比字段丰富重要得多。

**认知三：slug 是整个内容系统的身份证**

它连接 URL、路由、搜索、内链、SEO。改动 slug 的代价极高，需要专门的检查流程。

**认知四：分类是结构性的，标签是检索性的**

category 决定内容在站点中的归属，必须严格管控。tags 服务于搜索和关联，相对灵活但也需要防止同义词泛滥。

**认知五：构建链路决定了排查问题的思路**

理解"内容发现 → 解析 → 验证 → 聚合 → 索引 → 渲染"这条链路，才能准确定位问题出在哪个环节。

**认知六：draft 的影响是系统级的**

不只是"不显示在列表页"这么简单，而是需要在构建、索引、聚合、搜索等所有环节都正确过滤。

# 本篇练习

**练习 1：画出你项目的数据链路图**

不需要画得很漂亮，用文字列表就行：

**练习 2：列出你项目的合法分类清单**

打开你的项目，看一下现有内容文件都用了哪些 category。把它们整理成一个表格，写进 content-rules.md。如果发现有近义分类（如"政策解读"和"政策分析"），决定保留哪一个，统一掉。

**练习 3：检查你的 draft 过滤逻辑**

在你项目的内容查询逻辑中搜索 `draft`。确认以下几个地方都正确过滤了草稿：列表查询、分类聚合、标签聚合、搜索索引生成。如果有遗漏，修复它，然后把"draft 过滤要求"写进 data-schema.md。
