> 📌 本篇核心认知：CLAUDE.md 不是给人看的说明文档，而是给 Claude 用的项目级配置文件。理解这个区别，是用好 Claude Code 的第一步。

# 你是不是也在重复做这件事？

如果你正在用 Claude Code 开发一个内容型知识库项目——比如一个 Markdown 驱动的技术文档站、一个 FAQ 知识库、或者一个公众号素材管理系统——你大概率经历过这样的场景：

每次打开 Claude Code，开始一个新会话，你都要交代一遍项目背景：

`"这个项目是一个内容型知识库，用 Next.js + TypeScript 搭建，内容源文件全部在 content/ 目录下，   用 Markdown 写，每篇有 frontmatter，包含 title、slug、category、tags 这些字段。   构建的时候会先解析内容，生成索引，然后渲染页面。对了，slug 不能随便改，因为它直接关联 URL 和搜索索引……"   `

说完这段，Claude 终于"理解"了你的项目。然后你让它改一篇内容的分类字段，它改得很好。

第二天，新会话。一切归零。

你又要从头解释一遍。

这还不是最糟的。最糟的是——有时候你忘了交代某个关键约束（比如"slug 不能改"），Claude 就真的把 slug 改了，导致页面 URL 断裂、搜索索引失效、内链全部 404。

你花了半小时排查，才发现问题出在"你没提醒它"。

这个场景每天都在无数开发者的工作流里重复发生。

而 CLAUDE.md，就是为了终结这个问题而存在

# CLAUDE.md 到底是什么

大多数开发者第一次听到 CLAUDE.md，会本能地把它和 README.md 归为一类——"不就是个项目说明文件嘛"。

这个理解是错的。而且这个错误认知，是导致很多人用不好 CLAUDE.md 的根本原因。

## 它是配置文件，不是说明文档

CLAUDE.md 和 README.md 最本质的区别在于：

**README.md 是写给人看的。**

它假设读者是一个新加入项目的人类开发者，需要了解项目背景、怎么安装、怎么运行、架构是什么样。它追求的是"全面"和"可读性"。

**CLAUDE.md 是写给 Claude Code 用的。**

它假设读者是一个 AI 协作代理，需要在进入项目的第一秒就知道：

- 这个项目是什么
    
- 哪些东西能动，哪些不能动
    
- 做事的正确顺序是什么
    
- 哪些地方容易出错
    

它追求的不是"全面"，而是**高信号密度**——每一条信息都应该直接影响 Claude 的行为质量。

## 它进入的是 system prompt，不是普通对话

这是一个技术层面的关键区别。

当你在 Claude Code 项目根目录放一个 CLAUDE.md 文件，Claude 启动时会自动读取它，并把内容注入到**系统级提示词（system prompt）**中。

这意味着 CLAUDE.md 里的规则，在 Claude 的处理优先级中高于你在对话里临时说的话。

用一个不太严谨但好理解的类比：

`CLAUDE.md  →  公司规章制度（长期生效，优先级高）   对话中的指令  →  会议上的临时安排（当次有效，优先级较低）   `

当两者不冲突时，都会被遵守。当两者矛盾时，系统级规则更容易被优先执行。

这就是为什么一份写得好的 CLAUDE.md，能让你的整个工作流发生质变——你不再需要每次都在对话里重复交代规则，因为规则已经在系统层面"预装"好了。

## 三个核心作用

理解了 CLAUDE.md 的本质后，它的三个核心作用就很清晰了：

**作用一：项目记忆**

Claude 每次启动新会话都是"失忆"的。CLAUDE.md 让它在进入项目的一瞬间就恢复记忆——知道项目是什么、用什么技术、关键目录在哪里、常用命令有哪些。

对于内容型知识库项目，这个"记忆"尤其重要。因为内容项目的结构和规则往往不是从代码中就能直接推断出来的——比如 frontmatter 的字段含义、slug 的命名约定、分类体系的设计逻辑，这些都需要显式告知。

**作用二：操作边界**

CLAUDE.md 可以明确告诉 Claude"什么不能做"。

`## Notes   - 不要修改已发布内容的 slug   - 不要随意新增 frontmatter 字段   - 不要批量修改分类体系   `

这类"禁止性规则"在内容型项目中特别关键。因为内容文件看起来只是普通的 Markdown，Claude 很容易低估修改它们的连锁反应。一个 slug 的改动，背后牵连的是 URL、路由、搜索索引、聚合页、SEO 和内链——但 Claude 不会自动意识到这些。

你需要在 CLAUDE.md 里把这些边界画清楚。

**作用三：上下文预热**

没有 CLAUDE.md 的时候，Claude 的起点是"零认知"——它需要你用对话一点一点给它补充上下文，才能开始有效工作。

有了 CLAUDE.md，Claude 的起点变成了"已理解项目基础"——它一进来就知道目录结构、技术栈、关键规则和工作流程，可以直接进入"做事模式"。

这个差距，在简单任务上可能不明显。但在复杂任务上——比如"帮我新增一种内容类型"或"把这篇文章从 articles 迁移到 guides 并更新所有关联"——差距是巨大的。

# 为什么你的 CLAUDE.md 有时候"不管用"

很多开发者写了 CLAUDE.md，但发现 Claude 好像并没有严格遵守里面的规则。

这不是 bug，而是设计。

## Claude Code 的 system-reminder 机制

Claude Code 在读取你的 CLAUDE.md 后，不是直接把内容原样丢给 Claude，而是会用一层包装：

`<system-reminder>     IMPORTANT: this context may or may not be relevant to your tasks.      You should not respond to this context unless it is highly relevant to your task.   </system-reminder>   `

翻译成人话就是：**Claude 被提醒了——CLAUDE.md 里的内容不一定和当前任务有关，只有"高度相关"的部分才需要遵守。**

这意味着：

- 如果你写了 50 条规则，其中只有 5 条和当前任务相关，Claude 会重点参考那 5 条
    
- 如果你的 CLAUDE.md 全是泛泛而谈的内容，Claude 可能判断"没有高度相关的"，于是忽略大部分
    
- 如果不同规则之间存在矛盾或模糊地带，Claude 的行为可能不稳定
    

这就引出了 CLAUDE.md 写作中最重要的一个原则。

## "指令预算"概念

你可以把 Claude 对 CLAUDE.md 的处理能力想象成一个**有限预算**。

业界经验表明，前沿大语言模型能稳定遵循的指令数量大约在 150-200 条。超过这个范围，指令遵循质量不是断崖式下降，而是**均匀衰减**——所有规则都会被执行得差一些。

而 Claude Code 自身的系统提示词已经占用了大约 50 条指令的"预算"。

留给你的 CLAUDE.md，大概只有 100-150 条指令的空间。

但这里的重点不是让你去数条数。真正该记住的是：

> **每增加一条无关规则，都在稀释真正重要的规则的执行力度。**

所以，CLAUDE.md 的最佳写法不是"尽可能写全"，而是"只留最重要的"。

## 80% 原则

这是一个非常实用的筛选标准：

> **如果一条规则不是 80% 以上的 Claude Code 会话都需要遵守的，它就不应该出现在 CLAUDE.md 里。**

比如：

- "所有内容文件必须包含 frontmatter" → 几乎每次内容相关任务都会用到 → ✅ 放进去
    
- "slug 命名必须用小写英文加中划线" → 每次新增或修改内容都要遵守 → ✅ 放进去
    
- "部署到 Vercel 时要先检查环境变量" → 只在部署时用到 → ❌ 不放主文件，放 docs/deployment.md
    
- "React 组件的 props 类型定义要放在组件上方" → 代码风格细节 → ❌ 交给 linter
    

这个原则帮你把 CLAUDE.md 从一个"什么都往里塞的规则仓库"变成一个"只保留高价值信息的配置中枢"。

当然，有些低频但高风险的规则也值得放进去——比如"永远不要删除 content/ 目录下的文件"这种，虽然不是每次都触发，但一旦触发后果严重。80% 不是死规则，而是帮你建立"预算意识"的思维工具。

# 层级系统：从全局到目录的规则分层

CLAUDE.md 不是只能写一个文件。它支持多层级结构，让你把规则按作用范围分层管理。

## 三层结构


`┌────────────────────────────────────────────┐   │  Level 1：全局级                            │   │  ~/.claude/CLAUDE.md                       │   │  适用于所有项目的个人偏好和通用规则            │   ├────────────────────────────────────────────┤   │  Level 2：项目级                            │   │  project/CLAUDE.md                         │   │  适用于整个项目的核心规则                     │   ├────────────────────────────────────────────┤   │  Level 3：目录级                            │   │  project/content/CLAUDE.md                 │   │  project/tests/CLAUDE.md                   │   │  只在 Claude 访问该目录时才生效               │   └────────────────────────────────────────────┘   `

![[Pasted image 20260320002041.png]]

Claude 启动时，会按顺序读取这些文件：先全局，再项目级，最后是当前任务涉及的目录级。

## 在内容型知识库项目中怎么用

对于内容型项目，三层结构可以这样设计：

**全局级**（~/.claude/CLAUDE.md）

放你个人的通用偏好，比如：

`- 回答使用中文   - 代码注释使用中文   - 优先使用 TypeScript   `

**项目级**（project/CLAUDE.md）

放整个项目的核心规则，包括项目背景、技术栈、关键目录、常用命令、全局规范和工作流。这是最重要的一层，后面整本手册主要围绕这一层展开。

**目录级**（project/content/CLAUDE.md）

放只和内容管理相关的规则。比如：

`# Content Directory Rules      本目录包含项目的所有内容源文件。      ## 修改规则   - 不要修改已发布内容的 slug   - 不要随意新增 frontmatter 字段   - 修改 category 或 tags 前，先确认是否影响聚合页和搜索索引   - 新增内容请使用已有模板结构   `

目录级 CLAUDE.md 只在 Claude 处理该目录下的文件时才会被加载。这样做的好处是：

- 项目级文件保持精简
    
- 目录级规则不会干扰其他任务
    
- 不同目录可以有不同的约束
    
- 未来项目变大后更容易维护
    

## CLAUDE.md vs CLAUDE.local.md

还有一个实用的区分：

- `CLAUDE.md`：提交到 Git，团队共享
    
- `CLAUDE.local.md`：加入 .gitignore，个人使用
    

`.local.md` 适合放你个人的偏好、实验性规则、或者你不想影响团队的临时调整。

# 为什么内容型知识库项目格外需要 CLAUDE.md

在所有项目类型中，内容型知识库项目是最需要 CLAUDE.md 的一类。原因不复杂：

## 内容文件的"隐性复杂度"

在普通代码项目中，如果你改了一个函数的入参，编译器或 TypeScript 会立刻报错。反馈是即时的。

但在内容型知识库项目中，一篇 Markdown 文件看起来只是一段文本，但它实际上同时是：
![[Pasted image 20260320002324.png]]

`一篇 Markdown 内容文件，同时是：      ├── 详情页的数据源          → 渲染详情页   ├── 列表页的数据条目         → 生成列表、卡片   ├── 搜索索引的输入          → 被搜索引擎收录   ├── 分类聚合的单元          → 出现在分类页   ├── 标签聚合的单元          → 出现在标签页   ├── 路由系统的映射源         → slug 决定 URL   ├── SEO 的元数据来源        → title/description 影响搜索排名   └── 内链和相关推荐的依据      → 被其他内容引用   `

Claude 默认不知道这些。它看到一个 Markdown 文件，只会把它当"一篇文本"来处理。

所以当你让 Claude "帮我改一下这篇文章的分类"，它可能就直接改了——而不会意识到这个改动还会影响分类聚合页、搜索索引、导航结构。

CLAUDE.md 的作用，就是把这些"隐性连锁关系"变成"显性规则"，让 Claude 在改内容之前就知道要考虑什么。

## Claude 在内容项目里最常犯的错

根据实际使用经验，Claude 在内容型知识库项目中最容易犯以下几类错误：

**1. 随意新增 frontmatter 字段**

你让它"给这篇文章加一个摘要"，它可能不是用你已有的 `description` 字段，而是自己发明一个 `summary` 或 `abstract` 字段。结果页面渲染模板不识别这个字段，摘要根本不会显示。

**2. 改动 slug 不考虑后果**

你让它"优化一下这篇文章的 URL"，它把 slug 从 `teacher-cert-guide` 改成 `teacher-certificate-application-guide`。改完之后，旧 URL 全部 404，搜索引擎已收录的链接失效，其他文章里引用这篇文章的内链也断了。

**3. 分类和标签命名不一致**

你项目里的分类叫"政策解读"，Claude 新建一篇内容时用了"政策分析"。虽然语义接近，但聚合页只识别精确匹配的分类名，这篇文章就从分类页上"消失"了。

**4. 改一篇内容影响多个系统环节**

你让它"更新一下这篇文章的日期字段"，它改了日期的同时顺手调整了文件名。结果构建脚本找不到这个文件，列表页报错，搜索索引也挂了。

这些问题的共同点是：**Claude 不了解你项目的完整数据链路**。

而 CLAUDE.md 正是解决这个问题的正确工具。

# 一个常见误区：把 CLAUDE.md 写成 README

很多人写 CLAUDE.md 的方式，和写 README 完全一样——长段落的项目介绍、详细的技术选型说明、完整的安装步骤、全量的 API 文档……

这是一个自然但错误的倾向。

## README 思维 vs 配置思维

对比一下两种写法：

**README 思维（❌ 不适合 CLAUDE.md）**

![[Pasted image 20260320002517.png]]

`# 项目简介      本项目是一个面向中小学教师的知识库系统，旨在为教师提供便捷的考试信息查询、   政策解读和备考资料检索服务。项目始于 2024 年 3 月，由三人团队开发维护。   目前已积累超过 500 篇知识库内容，覆盖教师资格证、教师编制考试、特岗教师   招聘等多个领域。      ## 技术选型说明      我们选择 Next.js 作为前端框架，主要基于以下考虑：   1. 支持 SSG 和 SSR 混合渲染，适合内容站点   2. 内置路由系统，减少配置成本   3. 社区生态丰富，文档完善   4. Vercel 部署体验极佳      数据库方面，由于内容量不大且以静态内容为主……   （继续写了 2000 字）   `

**配置思维（✅ 适合 CLAUDE.md）**

![[Pasted image 20260320002536.png]]

``# Project Context      教师知识库系统。内容型静态站点，Markdown 驱动。   处理任务时，优先理解现有内容结构，再做增量修改。      ## Tech Stack   - Next.js + TypeScript   - 内容源：content/ 目录下的 Markdown 文件   - 构建：npm run build（会解析内容、生成索引、渲染页面）      ## Key Directories   - `content/articles/`：知识库文章   - `content/faq/`：常见问题   - `lib/`：内容解析和工具函数   - `app/`：页面路由      ## Standards   - 不修改已发布内容的 slug   - 不随意新增 frontmatter 字段   - 新增内容必须使用统一的 frontmatter 结构   - 修改分类或标签前，确认是否影响聚合页和搜索   ``

差距非常明显：

|维度|README 思维|配置思维|
|---|---|---|
|阅读对象|人类（新加入的开发者）|Claude（AI 协作代理）|
|追求的目标|全面、易读、背景完整|精准、高密度、直接影响行为|
|信息筛选标准|新人需要知道什么|Claude 每次做事前必须知道什么|
|篇幅|越详细越好|越精练越好|
|变更频率|较低（项目重大变化时更新）|较高（随工作流持续迭代）|

## 你的 CLAUDE.md 应该像什么

如果要找一个更贴切的类比，CLAUDE.md 不像 README，更像是**给实习生第一天上班时的工作备忘录**：

- 项目是做什么的（一句话）
    
- 代码在哪里、怎么跑
    
- 哪些东西绝对不能碰
    
- 做常见任务的标准步骤
    
- 容易踩的坑
    

你不会给实习生一份 50 页的公司介绍。你会给他一张 A4 纸，上面写着最关键的 20 条信息，让他在最短时间内可以开始工作。

CLAUDE.md 就是那张 A4 纸。

# 一个重要的延伸：CLAUDE.md 不应该独自承担一切

到这里，你可能已经有了一个新的疑问：

> "既然 CLAUDE.md 要精简，那那些被我'删掉'的信息去哪了？总不能不要吧？"

问得好。这恰恰是这本手册后面要展开讲的核心策略——**渐进披露和规则分层**。

简单预告一下思路：
![[Pasted image 20260320002717.png]]

`CLAUDE.md（主文件）     │     │  只放核心、高频、长期有效的规则     │     ├── docs/content-rules.md    ← 内容规范，处理内容任务时读取     ├── docs/data-schema.md      ← 数据结构，修改字段时读取     ├── docs/build-process.md    ← 构建流程，排查构建问题时读取     ├── docs/testing.md          ← 验证策略，需要检查时读取     ├── docs/deployment.md       ← 发布流程，准备上线时读取     └── docs/code-conventions.md ← 代码约定，涉及实现逻辑时读取   `

CLAUDE.md 是入口，docs/ 是专项细则库。Claude 按需读取，而不是全量灌入。

在主文件里，你只需要写一段引用：
![[Pasted image 20260320002734.png]]

``## Additional Documentation      处理以下任务前，请优先查看对应文档：   - 内容结构与 frontmatter：`docs/content-rules.md`   - 字段与数据实体：`docs/data-schema.md`   - 构建与索引流程：`docs/build-process.md`      仅在任务相关时读取对应文档，不必每次全部加载。   ``

这样做的好处是：

- 主文件保持精简（信号密度高）
    
- 复杂规则有专项文档承载（深度够）
    
- Claude 按需加载（不浪费上下文预算）
    
- 项目变大后也容易维护（扩展性好）
    

这个分层设计，是后续几篇手册会详细展开的内容。

# 还不止于此：CLAUDE.md + hooks + subagents

如果你已经理解了 CLAUDE.md 的本质，那你可以再往前想一步——

CLAUDE.md 只是"定义规则"。但规则定义了，还需要有人（或有东西）来"执行规则"。

这就是 Claude Code 的 hooks 和 subagents 能力发挥作用的地方。

**hooks**：在 Claude 执行特定动作的前后，自动触发检查。

比如你在 CLAUDE.md 里写了"提交前必须通过 lint"。如果只靠 Claude 自己记住，它偶尔会忘。但如果你设置一个 pre-commit hook，自动运行 `npm run lint`，那 Claude 就不需要"记住"这条规则——hook 会自动执行。

CLAUDE.md 告诉 Claude "规则是什么"，hook 负责"让规则自动执行"。

**subagents**：独立的 Claude 实例，处理隔离型任务。

比如你让 Claude 帮你做一次"内容结构一致性审查"——检查所有内容文件的 frontmatter 是否符合规范。这种任务不需要（也不应该）和你当前正在开发的功能共享上下文。用一个 subagent 来处理，它只读 CLAUDE.md 和 docs/content-rules.md，专注做审查，不会被其他上下文干扰。

三者的协作模型：

`┌─────────────────────────────────┐   │        CLAUDE.md 定义规则        │   │    "什么该做，什么不该做"          │   └──────────┬──────────────────────┘              │        ┌─────┴──────┐        │            │        ▼            ▼   ┌─────────┐  ┌──────────┐   │  hooks  │  │subagents │   │自动执行  │  │专项隔离   │   │机械检查  │  │独立任务   │   └─────────┘  └──────────┘   `

这本手册的最后一篇，会详细讲解这套组合用法。

这里先建立一个认知：**CLAUDE.md 不是孤立的一个文件，而是整个 AI 协作系统的入口和骨架。**

# 本篇核心认知回顾

在开始动手写你自己的 CLAUDE.md 之前，确保你已经理解了这些关键认知：

**认知一：身份定位**

CLAUDE.md 是配置文件，不是说明文档。它写给 Claude 看，不是写给人看。它追求高信号密度，不追求全面覆盖。

**认知二：系统级别**

CLAUDE.md 的内容进入 Claude 的 system prompt，优先级高于对话中的临时指令。但 Claude 也会判断相关性——无关内容不会被严格遵守。

**认知三：预算有限**

Claude 能稳定遵循的指令总量是有限的。每条无关规则都在稀释重要规则的执行力度。用 80% 原则做筛选。

**认知四：内容项目特殊**

内容型知识库项目中的 Markdown 文件不只是文本，而是页面、搜索、聚合、路由等多个系统环节的输入。Claude 不会自动意识到这些连锁关系，你需要在 CLAUDE.md 中显式告知。

**认知五：不是孤军作战**

CLAUDE.md 是入口，但不应该独自承载一切。它需要和 docs/ 文档、hooks、subagents 配合，形成完整的协作规则体系。
