# 第 90 章—2025 年 CSS 方案选型指南：Tailwind vs CSS Modules vs CSS-in-JS

在 2024、2025 年的当下，Tailwind CSS 确实已经成为 React 生态（甚至整个前端生态）中**最火热**的样式解决方案。很多新同学或者正在做技术选型的团队都会有同样的困惑：

> "Tailwind 既然这么好用，CSS Modules 和 CSS-in-JS 还有必要学吗？新项目还会用吗？"

这章我们就来深入分析一下当下的 CSS 方案格局，帮你理清学习重点和选型思路。

## 1. 为什么 Tailwind CSS 会“大火”？

Tailwind CSS 的胜利其实是 **Developer Experience (DX)** 的胜利。它解决了传统 CSS 开发的几个痛点：

1.  **取名困难症**：再也不用纠结是叫 `.container-wrapper` 还是 `.wrapper-container`。
2.  **上下文切换**：不需要在 `.js` 和 `.css` 文件之间反复横跳。
3.  **样式文件膨胀**：传统 CSS 很容易“只增不减”，Tailwind 的原子化特性让 CSS 体积可控。
4.  **设计系统约束**：强制你使用预定义的颜色、间距，避免了 `margin: 13px` 这种魔术数字，让 UI 更统一。

对于**应用开发（App Development）**来说，Tailwind 极大地提升了开发效率。

## 2. CSS Modules：依然是“中流砥柱”

虽然 Tailwind 很火，但 CSS Modules 绝对**没有死**，也不应该被忽视。它依然有很强的生命力，主要体现在以下场景：

### 核心优势

- **零学习成本**：只要会写标准 CSS 就会用。没有 Tailwind 那种 `pb-4`、`flex-col` 的记忆负担。
- **样式隔离**：通过 Hash 类名完美解决全局污染问题，这是它诞生的初衷，至今依然有效。
- **代码分离**：对于喜欢“关注点分离（Separation of Concerns）”的开发者，把样式从 JSX 中剥离出去依然是一种信仰。

### 适用场景

1.  **组件库开发（Component Library）**：
    如果你在开发一个通用的组件库（比如 Ant Design 这种），CSS Modules 往往比 Tailwind 更合适。因为你不想强迫你的用户必须安装 Tailwind 才能使用你的组件。
2.  **存量项目维护**：
    世界上有海量的项目是前几年写的，它们大量使用了 CSS Modules（或 SASS/LESS）。你需要读懂它们。
3.  **复杂的 CSS 动画/伪类**：
    虽然 Tailwind 支持 `hover:`、`before:`，但在写极其复杂的 CSS 动画或选择器逻辑时，标准 CSS 依然可读性更好。

**结论**：**CSS Modules 依然值得了解**，它是 CSS 方案的“保底”选择。

## 3. CSS-in-JS（Styled-components, Emotion）：跌落神坛？

CSS-in-JS（尤其是 Runtime CSS-in-JS）在几年前是 React 生态的“王者”，但最近几年确实面临巨大的挑战。

### 面临的挑战

1.  **运行时性能开销（Runtime Overhead）**：
    Styled-components 需要在浏览器端解析 JS、生成 CSS class、注入 `<style>` 标签。这会阻塞主线程，影响页面加载性能（LCP）。
2.  **React Server Components (RSC) 兼容性**：
    这是最致命的一击。Next.js 等框架在大力推行 RSC（服务端组件），而 Styled-components 这种依赖**运行时**上下文的库，在 RSC 中使用变得非常麻烦（需要标记 `use client`，无法在服务端组件直接运行）。

### 现状

- **新项目**：在 Next.js / Remix 等新架构项目中，**不推荐**首选 Runtime CSS-in-JS。
- **老项目**：依然大量存在。
- **重交互应用**：对于需要根据大量 JS 状态动态计算样式的场景（比如一个画图工具、低代码平台），CSS-in-JS 依然是最灵活的。

### 进化：Zero-runtime CSS-in-JS

为了解决性能问题，社区演化出了 **Zero-runtime（零运行时）** 的 CSS-in-JS，比如：

- **Panda CSS**
- **Vanilla Extract**
- **StyleX** (Facebook 出品)

它们保留了 CSS-in-JS 的写法体验（TypeScript 类型提示、组件化），但在**构建时（Build Time）**就把 CSS 提取成了静态的 `.css` 文件，运行时没有性能损耗。

**结论**：传统的 Styled-components **热度在下降**，但 CSS-in-JS 的**思想**（将样式作为代码处理）依然通过 Zero-runtime 方案在延续。

## 4. 总结：你应该怎么学？

回答你最初的问题：**“还有必要深入了解吗？”**

### 给 2025 年 React 开发者的建议：

1.  **Tailwind CSS**：**必须精通**。

    - 它是目前构建 Web 应用（尤其是 B2C、SaaS、Dashboard）的首选。
    - 它是“原子化 CSS”思想的集大成者。

2.  **CSS Modules**：**掌握基础即可**。

    - 你不需要专门“学”，只要你会 CSS，知道它是怎么配置的（Vite/Webpack 默认支持），知道怎么 import 进来怎么用 `styles.className` 就够了。
    - **面试/工作中大概率会遇到**老项目迁移，或者维护组件库，所以不能完全不懂。

3.  **CSS-in-JS**：**了解概念，不必深究源码**。
    - 知道 `styled.div` 这种写法。
    - 知道它背后的 `props` 传参动态样式能力。
    - 如果新项目要用，建议关注 **Panda CSS** 或 **StyleX** 这种新一代方案。

### 一张图总结选型

| 场景                       | 推荐方案                          | 理由                               |
| :------------------------- | :-------------------------------- | :--------------------------------- |
| **通用 Web 应用 (App)**    | **Tailwind CSS**                  | 开发快，体积小，生态好 (shadcn/ui) |
| **公共组件库 (Lib)**       | **CSS Modules** / Vanilla Extract | 依赖少，样式隔离，不强侵入         |
| **重度动态样式 (Dynamic)** | **CSS-in-JS** (StyleX/Panda)      | 类型安全，JS 变量控制样式          |
| **老项目维护**             | **看情况**                        | 尊重现有架构，别为了重构而重构     |

所以，放心大胆地去用 Tailwind，但也别把 CSS Modules 这种“老手艺”彻底丢掉，它们在特定领域依然是把好手。
